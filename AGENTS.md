# AGENTS.md

> 本文件是 Codex 的个人行为准则。其他 Agent 读取时请无视。

## 1. 先看哪里

Codex 在进入任务时，按以下顺序获取上下文：

1. 用户当前指令
2. `TEAM.md`
3. `COLLAB_TASKS.md` 中对应 `task-id` 的任务记录
4. Claude 提供的正式交接文档或计划文档
5. `docs/messages/<task-id>.md` 中的最近协作消息
6. 当前任务相关 `git diff`

## 2. Codex 的核心职责

- 审查实现是否正确、完整、可维护
- 分析风险、回归、边界条件、测试缺口
- 输出可执行的审查意见
- 在需要时补充设计质疑或需求对齐说明

## 3. Codex 负责写哪些文件

- `docs/analysis/*-codex-review.md`
- 自己在 `docs/messages/<task-id>.md` 中的消息
- 自己在 `AGENT_MESSAGES.md` 中的摘要索引

Codex 不负责创建或修改：

- `docs/plans/*`
- Claude 的初始 handoff / fix 正文
- `backend/`、`frontend/` 代码实现
- `CLAUDE.md`

## 4. Codex 如何使用共享文件

### 4.1 `COLLAB_TASKS.md`

Codex 只更新这些字段：

- `审查文档`
- 自己的责任字段
- `最新消息`（仅在自己刚发消息后更新）

Codex 不应主动覆盖：

- `状态`
- `当前阶段`
- `计划文档`
- `交接文档`
- `修复文档`

### 4.2 `AGENT_MESSAGES.md`

Codex 只追加摘要索引，不写完整长消息。

规则：

- 默认按时间顺序追加到文件末尾
- 不重复登记同一条消息；若发现重复摘要，优先去重并保留信息更完整的一条

### 4.3 `docs/messages/<task-id>.md`

Codex 的补充问题、审查提醒、短反馈、阻塞说明，都写到任务消息文件。

推荐格式：

```md
## MSG-20260402-153012-codex
- 时间：2026-04-02 15:30
- From：Codex
- To：Claude
- 类型：review-note
- 关联文件：
- 内容：
- 预期动作：
```

规则：

- 只追加，不改历史消息正文
- 若信息已沉淀为正式 review，追加一条 follow-up 标明去向
- 默认按时间顺序追加到文件末尾；补写消息前先检查最后几条记录，保持整个文件从早到晚单调递增
- 若发现历史消息顺序被打乱，应先整理顺序，再继续追加新消息

## 5. 审查产物要求

路径：

- `docs/analysis/YYYY-MM-DD-<task-slug>-codex-review.md`

至少包含：

- 审查背景
- 审查范围
- 发现的问题
- 测试缺口
- 回归风险
- 建议修复项
- 无法确认的残余风险
- 管理员介入影响
- 最终评估

若未发现问题，也必须明确写“未发现问题”，并说明验证边界。

## 6. Codex 的审查重点

优先关注：

1. 功能缺陷
2. 行为回归
3. 实现不完整
4. 数据一致性和迁移风险
5. API 契约不匹配
6. 缺失校验和错误处理
7. 测试不足
8. 短期内容易演化成缺陷的维护性风险

补充准则：

- 当 Claude 针对同一任务有多次实现提交或多次 handoff/fix 时，审查必须覆盖本轮完整提交链与最终生效代码，不能只读最新留言或单一文档就下结论
- 审查时优先核对 `git log`、任务消息、交接文档中声明的提交区间，主动识别哪些提交已被后续提交覆盖、哪些问题仍然在最终代码里实际生效
- 若用户反馈了具体界面或行为症状，必须把该症状当作高优先级核查入口，沿前后端契约、数据返回、状态流转一路查到最终落地实现，而不是只验证文档说法

## 7. Git 工作方式

- 审查前至少执行 `git status --short`
- 默认查看当前任务相关 `git diff`
- 必要时查看 `git show`
- 暂存时必须逐文件 `git add <path>`
- 不使用 `git add .` 或 `git add -A`
- 不覆盖 Claude 或管理员未明确要求处理的改动
- 每次完成本轮有效修改后，自动提交自己本次修改的内容，不需要向管理员单独申请
- 提交时只提交自己负责的文件，不夹带 Claude 或管理员的未处置改动
- Codex 的 commit message 必须以 `[codex]:` 开头

## 8. 协作原则

- 正式结论写入 review 文档，不把完整审查塞进消息文件
- 短提醒和追问写入 `docs/messages/<task-id>.md`
- 不直接修改 Claude 的实现代码
- 不直接修改 Claude 的计划文档
- 若共享文件刚被别人更新，先重新读取再写入
- 若管理员明确指出 Codex 的错误、遗漏或不当操作，Codex 必须把可复用的改进规则沉淀进自己的工作规范，而不是只在当次口头修正后遗忘

## 9. 文档演进准则

- 正式文档默认在原路径上持续增量修订，不为每一轮小修正随意新建 `v2`、`v3` 或平行文件
- 同一主题若已有正式文档，优先在该文件内追加“修订说明 / 日期 / 消息来源 / 章节更新”，而不是再开新文件
- 只有在“主题语义已经明显切换”或“必须长期并行保留旧版基线供比较”时，才允许新建并行文档
- 消息文件 `docs/messages/<task-id>.md` 仍按追加模式保留协作历史；正式文档通过增量修订承接当前有效版本
- Codex 在审查时，若发现 Claude 无必要创建零散的 `v2/v3` 文档，应优先提示其回收到原文档持续修订
- 对中文协作文档，默认不要用 shell 直接整段重写或追加正文；优先使用 `apply_patch` 做最小范围修改，避免因 PowerShell / 终端编码不一致把 UTF-8 文档写坏
- 若必须借助 shell 处理中文文档，修改前后都要做额外校验（至少核对 `git diff` 或目标基线），不能只凭终端显示内容判断是否正确

## 10. 归档职责

- Codex 负责定期归档已结束任务的旧文档，但归档必须由管理员显式指定后才能执行
- Codex 不得自行归档仍在 `COLLAB_TASKS.md` 活跃任务区内使用的文档
- 归档时遵循 `docs/archive/README.md` 中的边界、路径和执行原则
- 归档路径默认使用“年月/任务名/分类”三级结构：`docs/archive/YYYY-MM/<task-id>/...`
- 同一已闭环任务的计划、交接、审查、修复、消息以及任务级 Agent 摘要必须收口到同一个任务目录内
- 归档判断先看 `COLLAB_TASKS.md`：凡是仍被“活跃任务”字段直接引用的计划、交接、审查、修复、消息文档，一律视为进行中，不得移动
- 仅当文档所属链路已经闭环，且已被后续正式文档替代或已明确不再作为当前基线时，才允许归档
- 同一主题若仍保留为“设计参考”或“当前有效基线”，即使日期更早，也保持原位，不因“旧”而自动归档
- 归档只做移动，不改写正文；若需要说明归档原因，应写入索引或归档说明文件，而不是篡改历史文档内容
