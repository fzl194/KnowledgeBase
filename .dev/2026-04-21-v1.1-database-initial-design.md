# CoreMasterKB 1.1 数据库初版设计

- 日期：2026-04-21
- 作者：Codex
- 状态：讨论草案
- 目的：沉淀 1.1 阶段数据库边界、主链路表设计与 LLM Runtime 独立库方案，供后续拆分给 claude-mining / claude-serving / claude-llm 三方实现。

## 1. 总体原则

1. 资产库和 LLM Runtime 库分离。
2. Mining 写资产库，Serving 读资产库。
3. claude-llm 独占维护 LLM Runtime 库。
4. Asset DB 中不记录完整 LLM 调用日志，只保留弱引用。
5. canonical 相关旧表保留，但 1.1 不再作为主查询路径。

整体边界：

```text
Knowledge Asset DB
  Mining 写入
  Serving 读取
  负责文档、片段、关系、检索单元、发布版本

LLM Runtime DB
  claude-llm 服务独占写入
  Mining / Serving 通过 client/API 调用
  负责 LLM 任务、队列、prompt、调用记录、JSON 解析结果
```

## 2. Knowledge Asset DB 主链路

1.1 主链路建议为：

```text
asset_source_batches
  -> asset_publish_versions
  -> asset_raw_documents
  -> asset_raw_segments
  -> asset_raw_segment_relations
  -> asset_retrieval_units
  -> optional asset_retrieval_embeddings
```

说明：

- `asset_canonical_segments`
- `asset_canonical_segment_sources`

这两张旧表可保留，但 1.1 不作为主查询路径。

## 3. Asset 表设计

### 3.1 `asset_source_batches`

记录一次输入来源批次，例如用户上传一个文件夹、一个压缩包，或者后端导入一批资料。

| 字段 | 含义 |
|---|---|
| `id` | 批次 ID |
| `batch_code` | 批次编码，便于人读和追踪，例如 `batch_20260420_001` |
| `source_type` | 来源类型，例如 `folder_scan/user_upload/api_import/expert_authored/official_vendor/other` |
| `description` | 批次描述 |
| `created_by` | 谁创建的，用户、系统、管理员 |
| `created_at` | 创建时间 |
| `metadata_json` | 批次级扩展信息，例如上传入口、原始压缩包名、前端表单信息 |

设计说明：

- 这里不放产品、版本、网元外层字段。
- 如果用户声明“这批属于命令类”或“这批属于某网元某版本”，可以先放进 `metadata_json`。
- 后续 Mining 再把稳定信息下沉到文档或检索单元的 `scope_json/facets_json`。

### 3.2 `asset_publish_versions`

记录一次可发布的知识资产版本。

| 字段 | 含义 |
|---|---|
| `id` | 发布版本 ID |
| `version_code` | 版本编码，例如 `pv_20260420_001` |
| `status` | `staging/active/archived/failed` |
| `base_publish_version_id` | 基于哪个旧版本构建，可为空 |
| `source_batch_id` | 主要来源批次 |
| `description` | 本次发布说明 |
| `build_started_at` | 构建开始时间 |
| `build_finished_at` | 构建结束时间 |
| `activated_at` | 激活时间 |
| `build_error` | 构建失败原因 |
| `metadata_json` | 版本级扩展信息，例如 pipeline 版本、统计摘要、质量门控结果 |

关键约束：

```text
同一时间只能有一个 active publish version
Serving 默认只读 active version
Mining 只写 staging，检查通过后再激活
```

### 3.3 `asset_raw_documents`

记录原始文档。一个输入文件对应一条记录。

| 字段 | 含义 |
|---|---|
| `id` | 文档 ID |
| `publish_version_id` | 所属发布版本 |
| `document_key` | 文档稳定键。不同版本中同一文件应尽量保持一致 |
| `source_uri` | 后端实际读取位置，例如本地绝对路径、对象存储 URI |
| `relative_path` | 相对本次输入根目录的路径，前端上传文件夹后主要靠它保持稳定 |
| `file_name` | 文件名 |
| `file_type` | `markdown/txt/html/pdf/doc/docx/other` |
| `source_type` | 来源类型，可继承 batch，也可单独标注 |
| `title` | 文档标题，Mining 从文件名或一级标题推断 |
| `document_type` | 文档类型，例如 `command/feature/procedure/troubleshooting/expert_note/project_note/other` |
| `content_hash` | 原始文件内容 hash |
| `copied_from_document_id` | 如果采用物理快照，可指向上一版本复制来的文档 |
| `origin_batch_id` | 最初来自哪个 source batch |
| `created_at` | 创建时间 |
| `scope_json` | 文档适用范围，例如产品、版本、网元、项目、客户、环境等 |
| `tags_json` | 标签，例如 `命令/特性/故障/配置/专家经验` |
| `structure_quality` | 结构质量，例如 `markdown_native/plain_text_only/full_html/mixed/unknown` |
| `processing_profile_json` | 解析过程信息，例如 parser 名称、转换器版本、是否从 html 转 md |
| `metadata_json` | 其他扩展，例如 `parse_status/skip_reason/inferred_by` |

设计说明：

- `source_uri` 和 `relative_path` 两个字段都保留。
- `source_uri` 用于后端真实读取和审计。
- `relative_path` 用于跨环境保持稳定身份和前端展示。

### 3.4 `asset_raw_segments`

记录从文档中切出来的事实片段。它是事实源，不是检索入口。

| 字段 | 含义 |
|---|---|
| `id` | raw segment ID |
| `publish_version_id` | 所属发布版本 |
| `raw_document_id` | 所属文档 |
| `segment_key` | 文档内稳定片段键 |
| `segment_index` | 文档内顺序 |
| `section_path` | 章节路径，建议 JSON 数组字符串，例如 `["业务感知","告警信息"]` |
| `section_title` | 当前章节标题 |
| `block_type` | 结构类型：`paragraph/list/table/code/blockquote/html_table/raw_html/unknown` |
| `semantic_role` | 语义角色：`concept/parameter/example/note/procedure_step/troubleshooting_step/constraint/alarm/checklist/unknown` |
| `raw_text` | 原始片段文本 |
| `normalized_text` | 归一化文本，用于 hash 和基础匹配 |
| `content_hash` | 原文 hash |
| `normalized_hash` | 归一化文本 hash |
| `token_count` | 粗略 token 数 |
| `copied_from_segment_id` | 物理快照时指向上一版本复制来源 |
| `structure_json` | 表格、列表、代码块等结构化信息 |
| `source_offsets_json` | 行号、字符偏移、页码等来源定位 |
| `entity_refs_json` | Mining 抽取出的实体引用，例如命令、参数、特性、网元、术语 |
| `metadata_json` | 其他扩展信息 |

设计说明：

- `raw_segments` 尽量忠实保存原文事实。
- 不要为了某个查询意图把它改造成答案材料。
- Serving 可以利用 `entity_refs_json`，但不应强依赖其中某个子字段必然存在。

### 3.5 `asset_raw_segment_relations`

新增。记录 raw segment 之间的上下文关系。

这不是本体图，而是文档片段关系图，主要服务于召回后的上下文扩展。

| 字段 | 含义 |
|---|---|
| `id` | 关系 ID |
| `publish_version_id` | 所属发布版本 |
| `source_raw_segment_id` | 起点 raw segment |
| `target_raw_segment_id` | 终点 raw segment |
| `relation_type` | 关系类型 |
| `weight` | 权重，默认可为 `1.0` |
| `confidence` | 置信度，规则生成可为 `1.0`，弱规则或 LLM 可更低 |
| `distance` | 文档距离，例如相邻为 `1` |
| `metadata_json` | 关系来源、规则名称、额外说明 |

1.1 最小关系类型：

| `relation_type` | 含义 |
|---|---|
| `previous` | 前一个片段 |
| `next` | 后一个片段 |
| `same_section` | 同一章节 |
| `same_parent_section` | 同一父章节 |
| `section_header_of` | 某标题片段是某内容片段的章节上下文 |

后续可扩展：

```text
has_parameter
has_example
has_constraint
has_procedure_step
has_troubleshooting
references
elaborates
condition
contrast
```

### 3.6 `asset_retrieval_units`

新增。Serving 的主检索入口。

它不是去重结果，而是检索视图。一个 raw segment 可以生成多个 retrieval unit。

| 字段 | 含义 |
|---|---|
| `id` | retrieval unit ID |
| `publish_version_id` | 所属发布版本 |
| `unit_key` | 稳定键，用于幂等重建 |
| `unit_type` | 检索单元类型 |
| `target_type` | 指向对象类型 |
| `target_id` | 指向对象 ID |
| `title` | 检索单元标题 |
| `text` | 返回给 Agent 的主要文本 |
| `search_text` | 用于 FTS / BM25 / embedding 的检索文本 |
| `block_type` | 继承或生成的结构类型 |
| `semantic_role` | 继承或生成的语义角色 |
| `facets_json` | 动态过滤维度，例如领域、专题、产品、版本、网元、项目、上传标签 |
| `entity_refs_json` | 实体引用 |
| `source_refs_json` | 来源引用，至少能回到 raw document / raw segment |
| `llm_result_refs_json` | 如果由 LLM 生成或增强，记录 LLM task/result 引用 |
| `weight` | 静态权重 |
| `created_at` | 创建时间 |
| `metadata_json` | 扩展信息 |

`unit_type` 初版建议：

| `unit_type` | 含义 |
|---|---|
| `raw_text` | 直接基于 raw segment 原文 |
| `contextual_text` | 拼接章节、标题、实体后的上下文化文本 |
| `summary` | 章节或片段集合摘要 |
| `generated_question` | LLM 或规则生成的可能问法 |
| `entity_card` | 围绕实体聚合的检索卡片 |
| `table_row` | 表格或列表拆出的行级单元 |

`target_type` 初版建议：

```text
raw_segment
section
document
entity
synthetic
```

1.1 最小要求：

```text
每个 raw_segment 至少生成一个 contextual_text retrieval unit
```

示例：

```text
raw_text: 该特性无告警信息。
section_path: ["业务感知","告警信息"]
search_text: 业务感知 告警信息 该特性无告警信息。
```

### 3.7 `asset_retrieval_embeddings`

可选表。1.1 可以先建表但不强制实现向量检索。

| 字段 | 含义 |
|---|---|
| `id` | embedding ID |
| `publish_version_id` | 所属发布版本 |
| `retrieval_unit_id` | 对应 retrieval unit |
| `embedding_model` | 模型名称 |
| `embedding_provider` | provider 名称，例如 `openai/local/bge-m3` |
| `text_kind` | 向量化的是 `search_text/text/summary/generated_question` |
| `embedding_dim` | 向量维度 |
| `embedding_vector` | 向量；SQLite dev 可先不存或存 JSON，生产可用 pgvector |
| `content_hash` | 被向量化文本 hash |
| `created_at` | 创建时间 |
| `metadata_json` | 扩展信息 |

设计说明：

- embedding 建议挂在 `retrieval_units` 上，而不是直接挂在 `raw_segments` 上。
- 原因是 retrieval unit 才是检索视图，可以包含上下文化文本。

### 3.8 canonical 旧表处理策略

以下旧表保留：

- `asset_canonical_segments`
- `asset_canonical_segment_sources`

但 1.1 主路径不再依赖。

后续如果需要去重，建议改为旁路能力，例如：

```text
asset_duplicate_groups
asset_duplicate_group_members
```

这不是 1.1 必做项。

## 4. LLM Runtime DB

LLM Runtime DB 是 `claude-llm` 独立服务的数据库。

Mining 和 Serving 不直接写这些表，只通过 LLM Runtime client/API 提交任务。

主链路建议为：

```text
agent_llm_prompt_templates
agent_llm_tasks
agent_llm_requests
agent_llm_attempts
agent_llm_results
agent_llm_events
```

## 5. LLM Runtime 表设计

### 5.1 `agent_llm_prompt_templates`

记录 prompt 模板。建议保留模板表，否则后续无法解释为什么同一段文本生成了不同结果。

| 字段 | 含义 |
|---|---|
| `id` | 模板 ID |
| `template_key` | 模板键，例如 `mining.retrieval_unit.v1` |
| `template_version` | 模板版本 |
| `purpose` | 用途，例如 `query_rewrite/retrieval_unit_generation/summary/entity_extraction` |
| `system_prompt` | system prompt |
| `user_prompt_template` | user prompt 模板 |
| `expected_output_type` | `json_object/json_array/text` |
| `output_schema_key` | 预期输出 schema 名 |
| `output_schema_json` | JSON schema |
| `status` | `active/archived/draft` |
| `created_at` | 创建时间 |
| `metadata_json` | 扩展信息 |

说明：

- 第一版即使不做动态模板管理，也建议在调用记录中保存 `prompt_template_key/version`。

### 5.2 `agent_llm_tasks`

记录一次逻辑 LLM 任务，也是队列表。

| 字段 | 含义 |
|---|---|
| `id` | task ID |
| `caller_domain` | 调用方领域：`mining/serving/evaluation/admin` |
| `pipeline_stage` | 调用阶段 |
| `ref_type` | 业务关联对象类型 |
| `ref_id` | 业务关联对象 ID |
| `publish_version_id` | 关联知识资产版本，可为空 |
| `request_id` | Serving 在线请求 ID，可为空 |
| `idempotency_key` | 幂等键，防止重复提交 |
| `status` | `queued/running/succeeded/failed/dead_letter/cancelled` |
| `priority` | 优先级 |
| `available_at` | 可执行时间，用于延迟和重试 |
| `lease_expires_at` | worker claim 租约过期时间 |
| `attempt_count` | 已尝试次数 |
| `max_attempts` | 最大尝试次数 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `started_at` | 开始时间 |
| `finished_at` | 结束时间 |
| `metadata_json` | 扩展信息 |

`caller_domain` 示例：

```text
mining
serving
evaluation
admin
```

Mining 的 `pipeline_stage` 示例：

```text
semantic_role_enrichment
entity_extraction
retrieval_unit_generation
section_summary
generated_question
conflict_summary
```

Serving 的 `pipeline_stage` 示例：

```text
query_rewrite
intent_extraction
retrieval_planning
rerank
context_compression
answer_draft
```

### 5.3 `agent_llm_requests`

记录一次任务的模型请求输入。

| 字段 | 含义 |
|---|---|
| `id` | request ID |
| `task_id` | 所属 task |
| `provider` | 模型供应方，例如 `openai/anthropic/local/mock` |
| `model` | 模型名称 |
| `prompt_template_key` | prompt 模板键 |
| `prompt_template_version` | prompt 模板版本 |
| `messages_json` | 实际发送给模型的 messages |
| `input_json` | 结构化输入 |
| `params_json` | `temperature/max_tokens/timeout` 等参数 |
| `expected_output_type` | `json_object/json_array/text` |
| `output_schema_key` | 预期 schema 名 |
| `output_schema_json` | 可选 JSON schema |
| `created_at` | 创建时间 |
| `metadata_json` | 扩展信息 |

说明：

- 这里必须记录实际 `messages_json`，而不只是记录模板名。

### 5.4 `agent_llm_attempts`

记录每一次实际 provider 调用。一个 task 可能会多次重试。

| 字段 | 含义 |
|---|---|
| `id` | attempt ID |
| `task_id` | 所属 task |
| `request_id` | 所属 request |
| `attempt_no` | 第几次尝试 |
| `status` | `running/succeeded/failed/timeout/rate_limited` |
| `provider_request_id` | provider 返回的请求 ID |
| `raw_output_text` | 模型原始文本输出 |
| `raw_response_json` | provider 原始响应摘要 |
| `error_type` | 错误类型 |
| `error_message` | 错误信息 |
| `prompt_tokens` | prompt token 数 |
| `completion_tokens` | completion token 数 |
| `total_tokens` | 总 token 数 |
| `latency_ms` | 调用耗时 |
| `started_at` | 开始时间 |
| `finished_at` | 结束时间 |
| `metadata_json` | 扩展信息 |

说明：

- JSON 解析失败也不能丢掉 `raw_output_text`。

### 5.5 `agent_llm_results`

记录最终解析结果。

| 字段 | 含义 |
|---|---|
| `id` | result ID |
| `task_id` | 所属 task |
| `attempt_id` | 成功的 attempt |
| `parse_status` | `not_required/succeeded/failed/schema_invalid` |
| `parsed_output_json` | 解析后的 JSON |
| `text_output` | 文本输出 |
| `parse_error` | JSON 解析错误 |
| `validation_errors_json` | schema 校验错误 |
| `confidence` | 可选置信度 |
| `created_at` | 创建时间 |
| `metadata_json` | 扩展信息 |

说明：

- `raw_output_text` 放在 `attempts`。
- `parsed_output_json` 放在 `results`。
- 解析失败时不能伪装成空对象成功。

### 5.6 `agent_llm_events`

记录任务状态事件，便于排查队列和状态流转问题。

| 字段 | 含义 |
|---|---|
| `id` | event ID |
| `task_id` | 所属 task |
| `event_type` | `submitted/claimed/retried/succeeded/failed/cancelled/dead_letter` |
| `message` | 简短说明 |
| `metadata_json` | 事件上下文 |
| `created_at` | 创建时间 |

## 6. Asset DB 与 LLM Runtime DB 的关联方式

Asset DB 不直接外键引用 LLM DB。

原因：

- 两者是独立服务。
- 未来可能是不同数据库、不同部署方式。
- 强外键会把两个系统耦死。

建议只保留弱引用 JSON，例如：

- `asset_retrieval_units.llm_result_refs_json`
- `asset_raw_segments.metadata_json`
- `asset_raw_segment_relations.metadata_json`

引用示例：

```json
{
  "runtime": "claude-llm",
  "tasks": [
    {
      "task_id": "llm_task_001",
      "result_id": "llm_result_001",
      "pipeline_stage": "generated_question",
      "prompt_template_key": "mining.generated_question.v1"
    }
  ]
}
```

## 7. 三个员工的边界

| 员工 | 负责数据库部分 | 不应该做 |
|---|---|---|
| `claude-mining` | 写 `asset_source_batches/publish_versions/raw_documents/raw_segments/raw_segment_relations/retrieval_units` | 不直接写 LLM 表，不写 Serving 私有逻辑 |
| `claude-serving` | 读 active version 的 `asset_retrieval_units/raw_segments/raw_segment_relations/raw_documents` | 不写 asset 表，不依赖 canonical 主路径 |
| `claude-llm` | 维护 `agent_llm_*` 全套表、队列、provider、JSON 解析 | 不写 asset 表，不 import Mining/Serving 业务代码 |

## 9. 当前结论

1.1 的数据库主路径应明确为：

```text
Mining 生产 raw + relation + retrieval unit
Serving 检索 retrieval unit，再用 relation 扩展上下文
LLM Runtime 独立记录所有 LLM 调用
```

这版的目标不是一次把所有能力做完，而是先把数据库主干定稳，使后续：

- embedding
- hybrid retrieval
- rerank
- query rewrite
- context compression
- entity graph / ontology

都能在不推翻主模型的前提下继续演进。
