# CoreMasterKB 1.1 数据库完整方案

- 日期：2026-04-21
- 作者：Codex
- 状态：待评审
- 目标：给出一套当前可落地、同时支持未来演进的数据库架构说明，完整覆盖：
  - 整体数据库架构
  - 每张核心表包含哪些字段
  - 每个字段怎么来
  - 表与表之间如何流转
  - 运行链路如何闭环
  - 这套架构当前能达到什么效果、未来如何演进

---

## 1. 设计结论

这版方案的核心不是“换文件”，也不是“每条数据打版本号”，而是：

```text
稳定内容对象
  -> build（一次完整知识构建）
  -> release / publish（把某个 build 正式对外生效）
```

当前正式设计分成三库：

```text
asset_core
mining_runtime
agent_llm_runtime
```

其中：

- `asset_core` 是正式知识资产库，也是 Serving 查询的主库。
- `mining_runtime` 是挖掘运行态库，只记录 run、阶段、失败、重试。
- `agent_llm_runtime` 是独立 LLM 服务库，只记录 prompt、task、request、attempt、result、event。

这版方案明确放弃以下路线：

1. 放弃“每个 publish version 物理复制一整套资产表”
2. 放弃“给每张资产表每一行都挂 start/end version”
3. 放弃“用换 active 文件来替代正式 publish 语义”
4. 放弃“当前就把千人千面做成每用户一套独立知识库”
5. 放弃“把 Mining 运行态直接塞进资产表”

这版方案明确保留以下核心能力：

1. 内容不重复存储
2. publish 有正式数据库语义
3. Serving 查询边界清晰
4. 支持回退到旧 release
5. 支持未来增量复用未变内容
6. 支持未来长出个性化规则层

## 2. 先看整体架构

这套方案不是先从表开始，而是先从职责层开始。

### 2.1 三库职责

#### `asset_core`

负责回答：

```text
系统里有哪些知识内容
一次完整知识构建 build 采用了哪些内容
哪个 release 当前正式对外生效
```

Serving 只读这个库。

#### `mining_runtime`

负责回答：

```text
这次挖掘 run 到哪一步了
哪个文档失败了
哪个阶段失败了
能不能续跑
```

它不存正式知识资产。

#### `agent_llm_runtime`

负责回答：

```text
谁发起了 LLM 任务
用了什么 prompt 模板
发了哪些 request
尝试了几次
最终解析出了什么结果
```

它也不存正式知识资产。

### 2.2 `asset_core` 内部的三层结构

`asset_core` 不是一层表，而是三层：

```text
内容对象层
  -> build 层
  -> release / publish 层
```

#### 内容对象层

解决：

```text
知识内容本体是什么
```

包括：

- `asset_source_batches`
- `asset_documents`
- `asset_document_snapshots`
- `asset_raw_segments`
- `asset_raw_segment_relations`
- `asset_retrieval_units`

#### Build 层

解决：

```text
一次完整知识视图 build 采用哪些 document snapshot
```

包括：

- `asset_builds`
- `asset_build_document_snapshots`

#### Release / Publish 层

解决：

```text
哪个 build 当前正式对外生效
```

包括：

- `asset_publish_releases`

### 2.3 最总数据流

```text
输入资料
  -> source batch
  -> mining run
  -> document
  -> document snapshot
  -> raw segments
  -> raw segment relations
  -> retrieval units
  -> build
  -> release
  -> serving query
```

这条链里每一段的作用是：

| 阶段 | 作用 |
|---|---|
| `source batch` | 给一批输入资料一个批次身份 |
| `mining run` | 记录这次处理过程 |
| `document` | 定义逻辑文档是谁 |
| `document snapshot` | 定义这个逻辑文档当前一份不可变内容快照 |
| `raw segments` | 固化原始事实片段 |
| `relations` | 固化段落和上下文关系 |
| `retrieval units` | 固化可检索、可返回给 Agent 的检索单元 |
| `build` | 选择本次知识视图采用哪些快照 |
| `release` | 决定哪个 build 当前正式生效 |
| `serving query` | 只在 active release 对应的 build 范围内检索 |

## 3. 为什么是这套结构

### 3.1 为什么必须有 `document snapshot`

因为我们要同时满足：

1. 内容不重复存储
2. build / publish 有正式承载对象
3. 不给每条 segment / unit 打版本号
4. 支持未来增量
5. 支持未来回退
6. 支持未来多视图和个性化

如果没有 `document snapshot`，版本变化只能落在：

1. 每张表每一行  
   复杂度过高
2. 整库替换  
   publish 失去正式数据库意义

所以这层是必要的最小复杂度。

### 3.2 为什么 build 只选择 `document snapshot`

因为 build 的职责应该是：

```text
本次知识视图中，每个逻辑文档采用哪一份内容快照
```

它不应该直接选：

- segment
- relation
- retrieval unit

否则：

1. 枚举量会急剧膨胀
2. publish 清单会过大
3. 后续演进和回退都会复杂

把选择粒度放在 `document snapshot`，是当前最稳的折中。

### 3.3 为什么 publish 必须是 `release -> build`

因为 publish 的职责不是“日志”，也不是“换文件”，而是：

```text
把某个已经完成校验的 build 正式挂到某个发布通道
```

只有这样：

1. publish 才有正式数据库语义
2. 回退才自然成立
3. Serving 查询边界才清楚

## 4. 文档拆分说明

每一部分都按：

```text
这张表解决什么问题
字段有哪些

```

来讲。

## Part 1. 公共内容身份层

涉及表：

- `asset_source_batches`
- `asset_documents`
- `asset_document_snapshots`

这层先回答：

```text
输入是谁
文档是谁
文档这一份内容是什么
```

### 4.1 `asset_source_batches`

#### 4.1.1 这张表解决什么问题

它记录的是一批输入资料的身份，不是运行状态。

例如：

- 用户上传了一个文件夹
- 系统扫描了一个目录
- 管理员导入了一批压缩包

都应该先形成一个 `source_batch`。

#### 4.1.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | 批次 ID | 系统生成 UUID / ULID |
| `batch_code` | 便于人读的批次编码 | 创建批次时按规则生成 |
| `source_type` | 来源类型，如 `folder_scan/folder_upload/api_import/manual_seed` | 由触发入口给出 |
| `description` | 批次说明 | 调用方填写，可为空 |
| `created_by` | 创建者 | 用户、管理员或系统 |
| `created_at` | 创建时间 | 系统时间 |
| `metadata_json` | 批次级扩展信息 | 调用方和系统共同补充 |

#### 4.1.3 `metadata_json` 放什么

推荐放批次级默认信息：

```json
{
  "storage_root_uri": "file:///kb/input/2026-04-21/",
  "default_document_type": "command",
  "batch_scope": {
    "product": "CloudCore",
    "version": "V100R023C10"
  },
  "tags": ["coldstart", "commands"]
}
```

这些都是“批次默认值”，不是最终精确知识。

#### 4.1.4 往下怎么流

它会流向两处：

1. `mining_runs`
   - 标识本次 run 在处理哪一批输入
2. `asset_document_snapshots`
   - 标识某份文档快照来自哪个批次

所以关系是：

```text
source_batch
  -> mining_run
  -> document_snapshot
```

---

### 4.2 `asset_documents`

#### 4.2.1 这张表解决什么问题

这是逻辑文档身份表，不是文档内容表。

它回答：

```text
系统认为这是不是同一个逻辑文档
```

例如：

- `02_commands/05_add_apn.md`
- 它后续内容改了
- 或从 html 重新转成 md

只要我们认定它还是同一个逻辑文档，就应共享同一个 `document_id`。

#### 4.2.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | 逻辑文档 ID | 系统生成 UUID / ULID |
| `document_key` | 稳定文档键 | 由相对路径、外部业务键或约定规则生成 |
| `document_name` | 文档展示名 | 主标题、文件名或调用方指定 |
| `document_type` | 文档类型 | 调用方、批次默认值、目录规则或分类器推断 |
| `metadata_json` | 稳定级扩展信息 | 系统与业务侧补充 |
| `created_at` | 首次创建时间 | 系统时间 |

#### 4.2.3 `document_key` 怎么定

当前 M1 / 1.1 场景建议优先使用：

```text
相对输入根目录的路径
```

未来如果有更稳定的外部业务键，也可以切换为外部键。

`document_key` 的职责是：

```text
稳定标识“这还是不是同一个逻辑文档”
```

不是用来表达内容版本。

#### 4.2.4 `document_name` 怎么来

优先级建议：

1. 文档解析后的主标题
2. 调用方显式指定
3. 文件名去后缀

#### 4.2.5 `document_type` 怎么来

优先级建议：

1. 调用方明确指定
2. `source_batch.metadata_json.default_document_type`
3. 目录规则判断
4. 分类器或规则推断
5. `unknown`

#### 4.2.6 往下怎么流

一个逻辑文档会有多个内容快照：

```text
document
  -> document_snapshot*
```

---

### 4.3 `asset_document_snapshots`

#### 4.3.1 这张表解决什么问题

这是第一张真正的核心表。

它记录：

```text
某个逻辑文档，在某次输入下，对应的一份不可变内容快照
```

#### 4.3.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | 文档快照 ID | 系统生成 UUID / ULID |
| `document_id` | 属于哪个逻辑文档 | 关联 `asset_documents.id` |
| `source_batch_id` | 来自哪个输入批次 | 当前 `asset_source_batches.id` |
| `relative_path` | 相对输入根目录的路径 | 扫描文件时得到 |
| `source_uri` | 原始来源位置 | 本地绝对路径、对象存储 URI 或上游给定地址 |
| `mime_type` | 输入格式，如 `text/markdown/text/plain/text/html/application/pdf` | 扩展名、探测器或上游给定 |
| `content_hash` | 原始输入内容 hash | 对原始内容求 hash |
| `title` | 快照标题 | 主标题、上游元信息或文件名推断 |
| `scope_json` | 文档级适用范围 | 批次默认值 + 文档级规则/抽取结果 |
| `tags_json` | 文档级标签 | 批次默认标签 + 文档级标签 |
| `parser_profile_json` | 解析链元信息 | parser / converter / profile 记录 |
| `metadata_json` | 其他扩展信息 | 系统补充 |
| `created_at` | 快照创建时间 | 系统时间 |

#### 4.3.3 每个字段怎么来

##### `document_id`

流程：

1. 扫描文件
2. 计算 `document_key`
3. 查 `asset_documents`
4. 若存在则复用旧 `document_id`
5. 若不存在则创建新文档
6. 再创建 snapshot

##### `source_batch_id`

来自当前导入批次。

##### `relative_path`

来自扫描文件时的相对路径。

##### `source_uri`

是系统真实读取原始文件的位置。

##### `mime_type`

优先级：

1. 上游显式传入
2. 按扩展名推断
3. 内容探测器判断
4. 默认 `application/octet-stream`

##### `content_hash`

对“原始输入内容”求 hash，不是对解析后文本求 hash。

它的职责是判断这份输入内容是否和已有 snapshot 相同。

##### `scope_json`

来源：

1. `source_batch.metadata_json.batch_scope`
2. 文档路径规则
3. 文档解析或后续抽取补充

##### `tags_json`

来源：

1. 批次默认标签
2. 路径和目录规则
3. 文档内容或后续抽取

##### `parser_profile_json`

它是加工链路元信息，用于强回溯：

```json
{
  "parser": "markdown_parser_v1",
  "converter": "productdoc_to_md_v3",
  "input_format": "html",
  "normalized_format": "markdown",
  "profile": "md_m1"
}
```

#### 4.3.4 快照怎么复用

规则建议：

```text
同一个 document_id 下，
如果新输入内容的 content_hash 与已有 snapshot 相同，
则复用旧 snapshot；
如果不同，则创建新 snapshot。
```

#### 4.3.5 往下怎么流

后面的正式内容都应该挂在 `document_snapshot_id` 下：

```text
document_snapshot
  -> raw_segments
  -> raw_segment_relations
  -> retrieval_units
```

#### 4.3.6 小结

这一层明确了三件事：

| 表 | 负责什么 |
|---|---|
| `asset_source_batches` | 输入批次是谁 |
| `asset_documents` | 逻辑文档是谁 |
| `asset_document_snapshots` | 该文档当前这份内容是什么 |

## Part 2. 文档快照派生内容层

涉及表：

- `asset_raw_segments`
- `asset_raw_segment_relations`
- `asset_retrieval_units`

这层回答：

```text
一份 document snapshot 如何派生出正式可查询内容
```

### 5.1 `asset_raw_segments`

#### 5.1.1 这张表解决什么问题

它记录某个文档快照切出来的原始事实片段，是事实源。

Serving 最终不一定直接返回 raw segment，但所有正式知识内容都应能回到 raw segment。

#### 5.1.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | raw segment ID | 系统生成 UUID / ULID |
| `document_snapshot_id` | 来源文档快照 | 来自 `asset_document_snapshots.id` |
| `segment_key` | 快照内稳定片段键 | 由 segmentation 规则生成 |
| `segment_index` | 文档内顺序 | segmentation 顺序编号 |
| `section_path` | 章节路径 | 解析结构与标题栈生成 |
| `section_title` | 当前所属章节标题 | 当前标题节点 |
| `block_type` | 结构类型，如 `paragraph/list/table/code/blockquote/html_table/raw_html/unknown` | parser / structure 阶段识别 |
| `semantic_role` | 语义角色，如 `concept/parameter/example/note/procedure_step/constraint/unknown` | 规则、抽取器或 LLM enrichment |
| `raw_text` | 原始片段文本 | segmentation 输出 |
| `normalized_text` | 归一化文本 | normalization 输出 |
| `content_hash` | 原文 hash | 对 `raw_text` 求 hash |
| `normalized_hash` | 归一化 hash | 对 `normalized_text` 求 hash |
| `token_count` | 粗略 token 数 | tokenizer 或简单估算 |
| `structure_json` | 列表、表格、代码块等结构化信息 | parser / structure 阶段输出 |
| `source_offsets_json` | 行号、页码、字符偏移等 | parser / converter 记录 |
| `entity_refs_json` | 抽取出的实体引用 | 规则、抽取器或 LLM enrichment |
| `metadata_json` | 其他扩展信息 | 系统补充 |

#### 5.1.3 关键字段怎么来

##### `segment_key`

应保证在同一个 `document_snapshot` 内稳定。

可来自：

- 标题路径 + 顺序号
- block path
- parser 生成的结构位置键

##### `section_path`

来自解析结构时维护的标题栈，例如：

```json
["业务感知", "告警信息"]
```

##### `block_type`

链路是：

```text
原始文档
  -> parser / structure
  -> ContentBlock.block_type
  -> segmentation
  -> asset_raw_segments.block_type
```

##### `semantic_role`

来源优先级建议：

1. 明确结构规则
2. 实体和标题规则推断
3. LLM enrichment
4. `unknown`

##### `structure_json`

这是为了保留表格、列表、代码等结构化特征，而不是只留扁平文本。

例如表格转成结构化 list 时，结构信息要能被保留下来。

#### 5.1.4 往下怎么流

raw segment 会流向两处：

```text
raw_segment
  -> raw_segment_relations
  -> retrieval_units
```

---

### 5.2 `asset_raw_segment_relations`

#### 5.2.1 这张表解决什么问题

它记录 raw segment 之间的上下文关系。

它不是本体关系，不是知识图谱事实边，而是：

```text
段落之间的篇章关系、上下文关系、结构关系
```

这对召回后的上下文扩展非常关键。

#### 5.2.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | relation ID | 系统生成 UUID / ULID |
| `document_snapshot_id` | 所属文档快照 | 来自父 `document_snapshot` |
| `source_raw_segment_id` | 起点 segment | 关联 `asset_raw_segments.id` |
| `target_raw_segment_id` | 终点 segment | 关联 `asset_raw_segments.id` |
| `relation_type` | 关系类型 | 结构规则或后续推断 |
| `weight` | 权重 | 规则默认值或计算结果 |
| `confidence` | 置信度 | 规则默认值或模型输出 |
| `distance` | 文档距离 | 由位置关系计算 |
| `metadata_json` | 扩展信息 | 系统补充 |

#### 5.2.3 当前应支持的最小关系类型

```text
previous
next
same_section
same_parent_section
section_header_of
```

后续可以扩展：

```text
has_parameter
has_example
has_constraint
elaborates
references
condition
contrast
```

#### 5.2.4 字段怎么来

##### `relation_type`

当前主要来自文档结构和位置规则：

- 前后相邻段落 -> `previous` / `next`
- 同标题路径 -> `same_section`
- 同父标题路径 -> `same_parent_section`
- 标题段与内容段 -> `section_header_of`

##### `distance`

可用：

- 相邻为 1
- 同 section 内按索引差计算
- 跨 section 可取更大值

#### 5.2.5 往下怎么流

Serving 命中 retrieval unit 后，会通过 `source_refs_json` 找到相关 raw segment，再通过 relation 做上下文扩展。

也就是说：

```text
retrieval_unit hit
  -> source raw segments
  -> relation expansion
```

---

### 5.3 `asset_retrieval_units`

#### 5.3.1 这张表解决什么问题

这是 Serving 的主检索入口。

它不是去重结果，也不是 canonical 替代品，而是：

```text
适合被检索、被排序、被返回给 Agent 的检索封装层
```

#### 5.3.2 设计原则

当前实现可以做到：

```text
每个可检索 raw segment 至少生成一个 retrieval unit
```

但抽象上必须保留扩展性：

1. 一个 raw segment 可以生成多个 unit
2. 一个 unit 可以引用多个 raw segments
3. unit 不一定只指向 raw segment

#### 5.3.3 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | retrieval unit ID | 系统生成 UUID / ULID |
| `document_snapshot_id` | 来源文档快照 | 来自父 `document_snapshot` |
| `unit_key` | 稳定键 | builder 生成 |
| `unit_type` | 单元类型，如 `raw_text/contextual_text/summary/generated_question/entity_card/table_row` | retrieval builder 决定 |
| `target_type` | 指向对象类型，如 `raw_segment/section/document/entity/synthetic` | builder 决定 |
| `target_ref_json` | 指向目标的弱结构引用 | builder 生成 |
| `title` | 单元标题 | 标题、section 或摘要标题 |
| `text` | 返回给 Agent 的正文 | builder 生成 |
| `search_text` | 用于检索的文本 | builder 生成 |
| `block_type` | 结构类型 | 来自主 source segment 或 builder 推导 |
| `semantic_role` | 语义角色 | 来自主 source segment 或 builder 推导 |
| `facets_json` | 动态 facet | 批次、文档、抽取结果共同汇总 |
| `entity_refs_json` | 实体引用 | 来自 raw segment 或 enrichment |
| `source_refs_json` | 来源引用 | builder 生成 |
| `llm_result_refs_json` | LLM 弱引用 | 如有 LLM 参与生成则记录 |
| `weight` | 静态权重 | builder 默认值 |
| `metadata_json` | 扩展信息 | 系统补充 |
| `created_at` | 创建时间 | 系统时间 |

#### 5.3.4 关键字段怎么来

##### `unit_type`

当前 1.1 最低要求是：

```text
contextual_text
```

后续可以增加：

- `summary`
- `generated_question`
- `entity_card`
- `table_row`

##### `title`

一般来自：

1. 当前 section title
2. 结构化块标题
3. builder 生成的摘要标题

##### `text`

这是最终返回给 Agent 的主正文，不一定等于原始 raw text。

例如：

```text
原文: 该特性无告警信息。
章节: 业务感知 / 告警信息
```

可以生成：

```text
业务感知 / 告警信息：该特性无告警信息。
```

##### `search_text`

它是专门给检索用的，不要求和 `text` 一样。

例如可加入：

- section 路径
- 标题
- 主要实体
- 原文

##### `target_ref_json`

不用强绑单个 `target_id`，避免把设计锁死成“只能 join raw segment”。

推荐结构如：

```json
{
  "type": "raw_segment",
  "raw_segment_ids": ["seg_001"]
}
```

或：

```json
{
  "type": "section",
  "section_path": ["业务感知", "告警信息"]
}
```

##### `source_refs_json`

这是 Serving 和资产层的关键桥梁。

至少应能回溯到：

```json
{
  "document_snapshot_id": "snap_001",
  "raw_segment_ids": ["seg_001", "seg_002"],
  "section_paths": [["业务感知", "告警信息"]]
}
```

Serving 命中 unit 后，必须靠它下钻到 raw 和 relations。

##### `facets_json`

这是动态过滤和排序增强信号，不是硬依赖。

可包含：

```json
{
  "document_type": "command",
  "scope": {
    "product": "CloudCore",
    "version": "V100R023C10"
  },
  "tags": ["coldstart", "apn"]
}
```

#### 5.3.5 往下怎么流

这张表会直接进入 Build 层，因为 Serving 主查询就是针对这张表。

但 Build 本身不直接选择 unit，而是通过 `document_snapshot` 间接选择 unit。

所以关系是：

```text
document_snapshot
  -> retrieval_units
build
  -> document_snapshot selection
  -> implicitly selects retrieval_units under those snapshots
```

#### 5.3.6 小结

这一层明确了：

| 表 | 负责什么 |
|---|---|
| `asset_raw_segments` | 原始事实片段 |
| `asset_raw_segment_relations` | 段落与上下文关系 |
| `asset_retrieval_units` | 检索和返回给 Agent 的正式单元 |

## Part 3. Build 层

涉及表：

- `asset_builds`
- `asset_build_document_snapshots`

这一层回答：

```text
一次完整知识视图 build 到底是什么
```

### 6.1 `asset_builds`

#### 6.1.1 这张表解决什么问题

它记录一次完整知识构建的身份和状态。

build 不是 release，但 release 必须基于 build。

#### 6.1.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | build ID | 系统生成 UUID / ULID |
| `build_code` | 人可读构建编码 | 系统生成 |
| `status` | `building/validated/failed/published/archived` | build 生命周期状态 |
| `build_mode` | `full/incremental` | 本次构建策略 |
| `source_batch_id` | 触发批次 | 来自 `asset_source_batches.id` |
| `parent_build_id` | 基于哪个旧 build | 增量构建时可填 |
| `mining_run_id` | 对应哪个挖掘 run | 来自 `mining_runs.id` |
| `summary_json` | 构建摘要 | 构建完成后统计 |
| `validation_json` | 校验结果 | 校验阶段输出 |
| `created_at` | 创建时间 | 系统时间 |
| `finished_at` | 完成时间 | 构建完成时间 |

#### 6.1.3 字段怎么来

##### `status`

建议状态流转：

```text
building -> validated -> published
building -> failed
published -> archived
```

##### `build_mode`

当前 1.1 可以主要是：

```text
full
```

但字段必须预留 `incremental`。

##### `summary_json`

建议至少统计：

```json
{
  "documents_selected": 120,
  "snapshots_reused": 98,
  "snapshots_new": 22,
  "segments": 5321,
  "relations": 8110,
  "retrieval_units": 6210
}
```

##### `validation_json`

建议至少记录：

```json
{
  "has_documents": true,
  "has_retrieval_units": true,
  "checks": [
    {"name": "retrieval_units_non_empty", "passed": true}
  ]
}
```

#### 6.1.4 往下怎么流

它会流向：

```text
asset_build_document_snapshots
asset_publish_releases
```

---

### 6.2 `asset_build_document_snapshots`

#### 6.2.1 这张表解决什么问题

这是整个方案的第二张核心表。

它记录：

```text
在 build_X 里，
逻辑文档 D 采用的是 document snapshot S
```

这张表就是 build 的正式“清单表”。

#### 6.2.2 为什么必须有这张表

因为它一次性解决了 4 个问题：

1. build 有明确内容边界
2. publish 有正式对象
3. 不需要给每个 segment / unit 打版本号
4. 不需要复制全量正文和下游对象

#### 6.2.3 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `build_id` | 属于哪个 build | 关联 `asset_builds.id` |
| `document_id` | 哪个逻辑文档 | 关联 `asset_documents.id` |
| `document_snapshot_id` | 采用哪个快照 | 关联 `asset_document_snapshots.id` |
| `selection_status` | `active/removed` | build selection 结果 |
| `reason` | `add/update/retain/remove` | 构建规则决定 |
| `metadata_json` | 扩展信息 | 系统补充 |

推荐主键：

```text
(build_id, document_id)
```

#### 6.2.4 每个字段怎么来

##### `document_snapshot_id`

来自 build 选择逻辑：

- 如果文档内容没变 -> 复用旧 snapshot
- 如果文档内容变了 -> 选择新 snapshot

##### `selection_status`

表示在这个 build 里该逻辑文档是否生效：

- `active` = 本 build 采用这份快照
- `removed` = 本 build 中该文档被移除

##### `reason`

建议值：

- `add`
- `update`
- `retain`
- `remove`

它用于解释为什么 build 这次选了这个 snapshot。

#### 6.2.5 为什么 build 不直接选择 segment / unit

因为只要 build 选择了一个 `document_snapshot`，它下面的：

- `raw_segments`
- `raw_segment_relations`
- `retrieval_units`

就自然进入这个 build 的知识视图。

所以 build 只需要选到 snapshot 层即可。

#### 6.2.6 往下怎么流

它直接流向 publish：

```text
build
  -> build_document_snapshots
  -> release
```

## Part 4. Release / Publish 层

涉及表：

- `asset_publish_releases`

这一层回答：

```text
哪个 build 当前正式对外生效
```

### 7.1 `asset_publish_releases`

#### 7.1.1 这张表解决什么问题

这是第三张核心表。

它记录的是正式发布，而不是构建过程。

publish 的正式定义是：

```text
把某个已经完成校验的 build 挂到某个发布通道
```

#### 7.1.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | release ID | 系统生成 UUID / ULID |
| `release_code` | 人可读发布编码 | 系统生成 |
| `build_id` | 发布的是哪个 build | 关联 `asset_builds.id` |
| `channel` | 发布通道，如 `prod/test/preview` | 发布动作指定 |
| `status` | `staging/active/retired/failed` | release 生命周期状态 |
| `previous_release_id` | 上一个 release | 发布时查当前 channel active release |
| `released_by` | 谁发布 | 用户、管理员或系统 |
| `release_notes` | 发布说明 | 调用方填写 |
| `activated_at` | 激活时间 | 发布成功时间 |
| `deactivated_at` | 下线时间 | 被替换或撤销时填写 |
| `metadata_json` | 发布扩展信息 | 系统补充 |

#### 7.1.3 每个字段怎么来

##### `build_id`

来自已完成并校验通过的 build。

##### `channel`

发布动作指定，例如：

- `prod`
- `test`
- `preview`

##### `status`

建议状态流转：

```text
staging -> active -> retired
staging -> failed
```

##### `previous_release_id`

发布时查当前 channel 的 active release，如果存在则挂上。

#### 7.1.4 为什么它能支撑回退

因为 release 是数据库内正式对象。

回退时，不需要重跑整个 Mining，只需要：

```text
把某个旧 release 重新设回 active
```

前提是：

- 旧 release 仍然存在
- 它指向的 build 仍然存在

#### 7.1.5 Serving 怎么查

Serving 正确做法：

1. 查当前 channel 的 active release
2. 拿到 `build_id`
3. 查 `asset_build_document_snapshots`
4. 确定当前 build 采用哪些 snapshot
5. 只检索这些 snapshot 下的 `asset_retrieval_units`

所以 publish 不是日志，而是查询边界的正式控制点。

## Part 5. Mining Runtime 层

涉及表：

- `mining_runs`
- `mining_run_documents`
- `mining_stage_events`

这一层只记录过程，不记录正式知识视图。

### 8.0 Mining 的职责边界

这里必须明确：Mining 虽然是一条完整 pipeline，但设计上必须拆成两个阶段，不能写成一个混合 God Function。

#### Phase 1. Document Mining

输入：

- `source_batch`
- 输入文件集合

输出：

- `asset_documents`
- `asset_document_snapshots`
- `asset_raw_segments`
- `asset_raw_segment_relations`
- `asset_retrieval_units`
- 每个文档在 `mining_run_documents` 中的处理结果

它的职责是：

```text
把输入文档加工成稳定内容对象
```

#### Phase 2. Build & Publish

输入：

- 本轮 committed 的 snapshots
- 上一个 active release 对应的 build（如需复用旧 snapshot）

输出：

- `asset_builds`
- `asset_build_document_snapshots`
- `asset_publish_releases`

它的职责是：

```text
把稳定内容对象组装成可对外服务的知识视图，并正式发布
```

这两阶段分开后，才能同时做到：

1. 文档级失败不直接污染发布逻辑
2. build / publish 有独立可追溯事件
3. 后续可以分别优化文档挖掘与发布流程

### 8.1 为什么 runtime 不能塞进资产表

因为运行态和资产态是两种完全不同的东西：

| 类别 | 例子 |
|---|---|
| 运行态 | 正在处理、失败、重试、第几阶段 |
| 资产态 | 这是哪份快照、属于哪个 build、被哪个 release 采用 |

如果把它们混进同一张表，状态语义会混乱，Serving 也会被迫看到噪音状态。

---

### 8.2 `mining_runs`

#### 8.2.1 这张表解决什么问题

记录一次挖掘执行的总体身份和统计。

#### 8.2.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | run ID | 系统生成 UUID / ULID |
| `source_batch_id` | 处理的是哪一批输入 | 关联 `asset_source_batches.id` |
| `input_path` | 输入根目录 | 触发参数 |
| `status` | `running/completed/interrupted/failed` | run 生命周期状态 |
| `build_id` | 最终产出的 build，可为空 | 成功构建后回填 |
| `total_documents` | 本次发现文件总数 | 扫描统计 |
| `new_count` | 新增文档数 | 扫描与比对结果 |
| `updated_count` | 变更文档数 | 扫描与比对结果 |
| `skipped_count` | 跳过文档数 | 扫描与比对结果 |
| `failed_count` | 失败文档数 | 运行统计 |
| `committed_count` | 成功提交文档数 | 运行统计 |
| `started_at` | 开始时间 | 系统时间 |
| `finished_at` | 结束时间 | 系统时间 |
| `error_summary` | 失败摘要 | 运行结束生成 |
| `metadata_json` | 其他配置与统计 | 系统补充 |

#### 8.2.3 `metadata_json` 建议内容

建议至少包含：

```json
{
  "pipeline_profile": "m1_1_default",
  "resume_mode": "from_stable_point",
  "input_summary": {
    "root_path": "D:/kb/input"
  }
}
```

### 8.3 `mining_run_documents`

#### 8.3.1 这张表解决什么问题

记录每个逻辑文档在某次 run 中的处理状态。

#### 8.3.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | run-document ID | 系统生成 UUID / ULID |
| `run_id` | 属于哪个 run | 关联 `mining_runs.id` |
| `document_key` | 哪个逻辑文档 | 扫描阶段计算 |
| `content_hash` | 当前输入内容 hash | 扫描阶段计算 |
| `action` | `NEW/UPDATE/SKIP/REMOVE` | 与已有 snapshot/build 比对得出 |
| `status` | `pending/processing/committed/failed/skipped` | 运行状态 |
| `document_id` | 对应逻辑文档 ID，可为空 | 查找或新建后回填 |
| `document_snapshot_id` | 产出的 snapshot，可为空 | 快照创建或复用后回填 |
| `error_message` | 失败原因 | 失败时记录 |
| `started_at` | 开始处理时间 | 系统时间 |
| `finished_at` | 结束时间 | 系统时间 |
| `metadata_json` | 扩展信息 | 系统补充 |

#### 8.3.3 `status` 语义

建议值：

- `pending`
- `processing`
- `committed`
- `failed`
- `skipped`

含义：

- `pending`：已发现，尚未进入处理
- `processing`：处理中，可能存在未完成副作用
- `committed`：文档级稳定内容对象已经完整落地
- `failed`：本轮处理失败
- `skipped`：判定为无需处理，例如复用旧 snapshot

#### 8.3.4 `committed` 的正式定义

一个 `run_document` 只有在下列对象都已稳定落地后，才允许标记为 `committed`：

1. `asset_documents` 已创建或复用
2. `asset_document_snapshots` 已创建或复用
3. 当前 snapshot 对应的 `asset_raw_segments` 已完整落地
4. 当前 snapshot 对应的 `asset_raw_segment_relations` 已完整落地
5. 当前 snapshot 对应的 `asset_retrieval_units` 已完整落地

也就是说：

```text
committed = 文档级稳定内容对象已准备好
```

它不等于：

```text
该内容已经被 build 采用
或
该内容已经发布为 active release
```

这两个动作属于 Build & Publish 阶段。

---

### 8.4 `mining_stage_events`

#### 8.4.1 这张表解决什么问题

记录文档处理的阶段事件，用于诊断和统计。

#### 8.4.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | event ID | 系统生成 UUID / ULID |
| `run_id` | 属于哪个 run | 关联 `mining_runs.id` |
| `run_document_id` | 属于哪个 run-document | 关联 `mining_run_documents.id` |
| `stage` | 阶段名 | 当前 pipeline 阶段 |
| `status` | `started/completed/failed` | 阶段事件状态 |
| `duration_ms` | 耗时 | 阶段统计 |
| `output_summary` | 阶段输出摘要 | 系统记录 |
| `error_message` | 失败原因 | 阶段失败时记录 |
| `created_at` | 事件时间 | 系统时间 |

#### 8.4.3 `run_id` 与 `run_document_id` 的关系

这里必须同时支持两类事件：

##### 文档级阶段事件

例如：

- `parse`
- `segment`
- `enrich`
- `build_relations`
- `build_retrieval_units`

这些事件应满足：

- `run_id` 必填
- `run_document_id` 必填

##### Run 级阶段事件

例如：

- `select_snapshot`
- `assemble_build`
- `validate_build`
- `publish_release`

这些事件应满足：

- `run_id` 必填
- `run_document_id` 为空

也就是说：

```text
mining_stage_events 不能只挂 run_document_id，
还必须显式挂 run_id。
```

否则 run 级阶段没有自然归属。

#### 8.4.4 推荐 `stage` 值

```text
parse
segment
enrich
build_relations
build_retrieval_units
select_snapshot
assemble_build
validate_build
publish_release
```

#### 8.4.5 为什么要有 run 级阶段事件

因为 Build & Publish 阶段不是某个单独文档的阶段，而是整次 run 的公共阶段。

例如：

- 组装 build
- 校验 build
- 发布 release

这些动作如果只写日志、不入库，会导致：

1. 无法统一追溯 run 后半段发生了什么
2. 无法统计 build / publish 耗时
3. 无法精确判断是构建失败还是发布失败

### 8.5 断点续跑与恢复规则

这里要明确：v1.1 支持断点续跑，但不追求极细粒度的阶段内恢复。当前采用“回到最近稳定点”的策略。

#### 8.5.1 run 级恢复入口

恢复时：

1. 找到 `mining_runs.status in (interrupted, failed)` 的 run
2. 读取该 run 下全部 `mining_run_documents`
3. 按每个文档当前 `status` 决定恢复策略

#### 8.5.2 文档级恢复规则

##### `committed`

直接跳过。

因为它已经生成了稳定内容对象。

##### `pending`

正常进入处理。

##### `skipped`

直接跳过。

##### `processing` / `failed`

这里不做“从最后一个细粒度阶段直接拼接继续”，而是：

```text
先清理该 run 本次未完成副作用
再从最近稳定点恢复
```

#### 8.5.3 当前推荐的稳定点策略

对于未 `committed` 的文档，v1.1 推荐的保守策略是：

1. 保留已确定复用的 `asset_documents`
2. 如果本轮已创建新 `document_snapshot` 且未 committed，则删除该 snapshot 下本轮生成的：
   - `asset_raw_segments`
   - `asset_raw_segment_relations`
   - `asset_retrieval_units`
3. 必要时删除该未完成 snapshot 本身
4. 从 `segment` 阶段重做

也就是说，恢复策略不是：

```text
细粒度拼接半成品继续跑
```

而是：

```text
回到文档级最近稳定点后重做
```

这样做的原因是：

1. 简单
2. 稳定
3. 不容易留下脏半成品

#### 8.5.4 为什么不在 snapshot 上加 `mining_status`

当前不建议给 `asset_document_snapshots` 加：

```text
draft / complete
```

原因是：

1. 这会把运行态状态污染进资产层
2. 后续 raw / relation / unit 也会想跟着加状态
3. 资产层和 runtime 层边界会被重新打乱

当前正式规则是：

```text
snapshot 是否处于完整、可恢复、已提交状态，
由 mining_runtime 判断；
snapshot 是否正式可被 Serving 看到，
由 build / release 判断。
```

### 8.6 孤儿数据清理

这里也要明确：随着新 snapshot 产生，旧 snapshot 及其下游对象可能不再被任何 build 引用。

当前判断：

- 这是一个真实问题
- 但不是 v1.1 主链路的阻塞项

#### 8.6.1 为什么当前不阻塞

因为 Serving 是按：

```text
active release
  -> build
  -> selected snapshots
```

来查的。

所以即使存在孤儿 snapshot / segment / retrieval_unit，它们也不会被误当成 active 内容返回。

#### 8.6.2 当前文档中的正式定位

把孤儿清理定义为：

```text
后置治理项
```

不在 v1.1 主发布链路内同步执行。

#### 8.6.3 后续建议机制

后续可以增加一个 GC 任务，定期找出：

1. 不被任何 build 引用的 `asset_document_snapshots`
2. 再级联到其下的：
   - `asset_raw_segments`
   - `asset_raw_segment_relations`
   - `asset_retrieval_units`

并在保守条件下清理，例如：

- 老于某个保留窗口
- 不被任何 active / recent release 引用

### 8.7 Mining Runtime 三张表如何协同运作

这一节不再讲“字段定义”，而是讲它们在一次真实 run 里怎么配合。

#### 8.7.1 三张表各自扮演什么角色

##### `mining_runs`

这是整次挖掘任务的总记录。

它回答的是：

```text
这次 run 是谁
处理哪一批输入
当前整体状态如何
最终产出了哪个 build
```

##### `mining_run_documents`

这是这次 run 里每篇文档的状态机。

它回答的是：

```text
这篇文档这次是新增、更新、跳过还是移除
当前处理到哪一步
最终落成了哪个 document / snapshot
```

##### `mining_stage_events`

这是事件流水，不是状态快照。

它回答的是：

```text
某个阶段何时开始、何时完成、是否失败、耗时多少、输出了什么
```

三者关系可以概括成：

```text
mining_runs           = 一次 run 的总控制面
mining_run_documents  = 每篇文档的处理状态机
mining_stage_events   = 阶段事件流水
```

#### 8.7.2 一次 run 的完整流转

##### Step 1. 创建 `source_batch`

先在 `asset_core` 中创建：

```text
asset_source_batches
```

表示这次输入资料是哪一批。

##### Step 2. 创建 `mining_runs`

Mining 启动后，先在 `mining_runtime` 创建一条 `mining_runs`：

- `status = running`
- `source_batch_id = 当前 batch`
- `input_path = 输入目录`

这一步只是声明：

```text
一次新的挖掘 run 开始了
```

##### Step 3. 扫描输入文件并生成 `mining_run_documents`

对输入目录递归扫描后，每个文件生成一条 `mining_run_documents`。

这里会先做最轻量的判断：

1. 计算 `document_key`
2. 计算当前输入的 `content_hash`
3. 查已有 `asset_documents`
4. 查已有 `asset_document_snapshots`
5. 判断动作：
   - `NEW`
   - `UPDATE`
   - `SKIP`
   - `REMOVE`（后续如果引入显式移除逻辑）

这时 `mining_run_documents` 记录的是：

```text
这篇文档在本次 run 中应该怎么处理
```

但还没有正式生成新的 snapshot 或下游对象。

##### Step 4. 进入文档级处理

对于每条 `action != SKIP` 的文档：

- `mining_run_documents.status -> processing`
- 同时开始写 `mining_stage_events`

文档级阶段通常是：

```text
parse
segment
enrich
build_relations
build_retrieval_units
```

每个阶段开始/完成/失败都要写事件。

##### Step 5. 文档级内容对象落地到 `asset_core`

当某个文档文档级处理成功后，会写入正式内容对象：

- `asset_documents`
- `asset_document_snapshots`
- `asset_raw_segments`
- `asset_raw_segment_relations`
- `asset_retrieval_units`

然后在 `mining_run_documents` 中回填：

- `document_id`
- `document_snapshot_id`

最后把该文档标记为：

```text
status = committed
```

这里必须强调：

```text
committed != published
```

`committed` 只表示：

```text
文档级稳定内容对象已经落地
```

不表示：

```text
它已经进入当前 active release
```

##### Step 6. 进入 run 级阶段

当本次 run 中所有文档级处理完成后，进入 run 级公共阶段：

```text
select_snapshot
assemble_build
validate_build
publish_release
```

这些阶段的事件也写入 `mining_stage_events`，但这里：

- `run_id` 必填
- `run_document_id = NULL`

因为这些阶段属于整次 run，不属于某一篇文档。

##### Step 7. 组装 build

Mining 基于：

- 本轮 committed 的 snapshots
- 上一个 active release 对应 build 的已选 snapshots（如果要复用未变内容）

组装出：

- `asset_builds`
- `asset_build_document_snapshots`

然后把 `mining_runs.build_id` 回填为本次产出的 build。

##### Step 8. 校验与发布

如果 build 校验通过，则写：

- `asset_publish_releases`

发布动作完成后：

- 旧 active release -> `retired`
- 新 release -> `active`

最后：

- `mining_runs.status -> completed`

如果 build 或 publish 失败：

- `mining_runs.status -> failed`
- 当前 active release 保持不变

#### 8.7.3 状态语义总结

##### `mining_runs.status`

表示整次 run 的总体状态，例如：

- `running`
- `completed`
- `interrupted`
- `failed`

##### `mining_run_documents.status`

表示单篇文档在本次 run 中的状态，例如：

- `pending`
- `processing`
- `committed`
- `failed`
- `skipped`

##### `mining_stage_events`

不承载“当前状态”，而承载“阶段事件历史”。

所以它是：

```text
event log
```

不是：

```text
current state table
```

### 8.8 Mining Runtime 与资产表的协同关系

这一节回答两个最容易混淆的问题：

1. runtime 表是不是在替代资产表
2. runtime 表和资产表的边界到底在哪里

#### 8.8.1 `mining_runs` 和资产表的关系

`mining_runs` 不拥有正式知识内容，它只记录：

- 这次 run 对应哪一批输入
- 这次 run 最终产出了哪个 build

它和资产层最关键的连接点是：

- `source_batch_id`
- `build_id`

所以它是：

```text
过程入口 + 过程出口
```

不是内容表。

#### 8.8.2 `mining_run_documents` 和资产表的关系

这是文档处理过程和正式内容对象之间的桥。

它最终会连到：

- `asset_documents`
- `asset_document_snapshots`

并且通过 `document_snapshot_id` 间接对应到：

- `asset_raw_segments`
- `asset_raw_segment_relations`
- `asset_retrieval_units`

它的作用是：

```text
告诉你：这次 run 里某篇文档最后落成了哪个逻辑文档、哪个快照
```

#### 8.8.3 `mining_stage_events` 和资产表的关系

它本身不拥有资产。

它的作用是解释：

- 哪些资产是在什么阶段生成的
- 哪一步失败了
- 是文档级失败还是 build / publish 级失败

所以它是：

```text
过程追溯表
```

不是：

```text
资产归属表
```

#### 8.8.4 最关键的两个边界

##### 边界 1：`committed != published`

这条必须单独强调。

```text
mining_run_documents.status = committed
```

只表示：

```text
文档级内容对象已经完整落地
```

不表示：

```text
这个 snapshot 已经被 build 采用
或
已经被 active release 对外发布
```

##### 边界 2：snapshot 存在 != Serving 可见

资产库中存在一个 `asset_document_snapshots`，不代表 Serving 就能查到它。

Serving 只能看到：

```text
active release
  -> build
  -> build 采用的 snapshots
  -> 这些 snapshots 下的 retrieval_units
```

也就是说：

```text
有 snapshot
!=
已发布可见
```

#### 8.8.5 三层协同的最终总结

可以用一句话收束：

```text
runtime 负责“怎么生产出来”，
asset 内容层负责“生产出了什么”，
build / release 负责“哪些正式生效”。
```

这三层不能互相替代。

## Part 6. LLM Runtime 层

这层由独立 `claude-llm` 服务维护。

涉及表：

- `agent_llm_prompt_templates`
- `agent_llm_tasks`
- `agent_llm_requests`
- `agent_llm_attempts`
- `agent_llm_results`
- `agent_llm_events`

### 9.1 为什么必须独立

因为 LLM 运行态是独立的：

1. Mining 和 Serving 都可能调用它
2. prompt 模板、重试、失败、结构化解析都要统一治理
3. 不能让 Mining 和 Serving 各自私建一套 LLM 调用表

### 9.2 各表职责

#### `agent_llm_prompt_templates`

记录 prompt 模板与 schema 契约。

#### `agent_llm_tasks`

记录一次逻辑 LLM 任务。

#### `agent_llm_requests`

记录实际发送给模型的 request。

#### `agent_llm_attempts`

记录每次 provider 调用尝试。

#### `agent_llm_results`

记录解析后的最终结构化结果。

#### `agent_llm_events`

记录任务状态事件。

### 9.3 和资产库怎么关联

这里不做强外键，而做弱引用。

`asset_retrieval_units.llm_result_refs_json` 可以记录：

```json
{
  "runtime": "agent_llm_runtime",
  "tasks": [
    {
      "task_id": "task_001",
      "result_id": "result_001",
      "pipeline_stage": "generated_question"
    }
  ]
}
```

这样既能回溯，又不会把资产表和 LLM 运行态紧耦合。

## 11. 当前这套架构能达到什么效果

### 11.1 publish 合理

publish 不再是：

- 记日志
- 换文件
- 事后补救

而是：

```text
把某个已验证 build 正式挂到 channel
```

### 11.2 不重复存储大对象

文档内容没变，就复用旧 snapshot，不重新复制：

- 文档正文
- segment
- relation
- retrieval unit

### 11.3 支持回退

如果旧 release 仍在，直接切回旧 release 即可。

### 11.4 支持未来增量

即使 1.1 一开始先 full build，后面也可以按：

- `document_key`
- `content_hash`

做 snapshot 复用和增量构建。

### 11.5 支持未来个性化

当前不直接做“每用户一套知识库”。

未来做法应该是：

```text
共享 asset_core
  -> 公共 build / release
  -> 个性化规则层
  -> 查询时做 filtering / boosting
```

也就是说，当前数据库架构的目标是：

```text
不要堵死未来个性化的路
```

### 11.6 Serving 查询边界清晰

Serving 不看：

- mining runtime
- 半成品 build
- 失败 run
- LLM attempt

Serving 只看：

- active release
- 它对应的 build
- build 选中的 snapshot 下的 retrieval units

## 13.1 修订说明：共享内容快照模型

以下内容是对本文前面 `document snapshot` 语义的正式修订。若与前文某些“文档专属快照”的表述冲突，以本节为准。

### 13.1.1 修订原因

前文较早的描述默认了：

```text
一个 document
  -> 多个 document_snapshot
一个 snapshot
  -> 只属于一个 document
```

这个模型能工作，但会导致一个问题：

- 文档 A 和文档 C
- 如果规范化后正文完全一致
- 仍然会重复生成两份 snapshot，以及两套下游 segments / relations / retrieval_units

这不符合“内容不重复存储”的目标。

因此这里正式修订为：

```text
document = 逻辑文档身份
snapshot = 可共享的内容快照
document 和 snapshot 之间通过映射关系关联
```

### 13.1.2 修订后的关系模型

修订后的正式关系应理解为：

```text
asset_documents
  -> asset_document_snapshot_links
  -> asset_document_snapshots
```

语义是：

- 一个 document 可以在历史上引用多个 snapshots
- 多个 documents 也可以共享同一个 snapshot

也就是说：

```text
一个 document -> 多个 snapshots
多个 documents -> 同一个 snapshot
```

都是允许的。

### 13.1.3 `asset_document_snapshots` 的新语义

前文中的 `asset_document_snapshots`，现在应统一理解为：

```text
共享内容快照表
```

它不再是“某个 document 专属的一版内容”，而是：

```text
一份可被一个或多个 documents 共享引用的不可变内容对象
```

因此，前文涉及 `asset_document_snapshots` 的地方，都应按下面这条来理解：

```text
snapshot 的唯一性主要由内容决定，而不是由 document_id 决定
```

### 13.1.4 Snapshot 复用规则的修订

原先隐含的复用规则是：

```text
同一个 document 下，
如果 content_hash 相同，则复用旧 snapshot
```

现在修订为：

```text
对规范化正文求 normalized_content_hash；
如果全库已有相同 hash 的 snapshot，则复用；
如果没有，则创建新的 snapshot。
```

这意味着：

1. 同一文档跨批次没变，可复用旧 snapshot
2. 不同文档只要内容完全一致，也可共享同一份 snapshot

### 13.1.5 为什么还保留 `asset_documents`

因为：

- `snapshot` 解决的是“内容是什么”
- `document` 解决的是“文档身份是谁”

这两件事不是一回事。

例如：

- 产品文档 A
- 产品文档 C

即使它们暂时引用同一份 5G 介绍 snapshot，它们仍然是两个不同的逻辑文档。

所以：

```text
document 身份不能被 snapshot 取代
```

### 13.1.6 新增映射表：`asset_document_snapshot_links`

为了支撑“逻辑文档身份”和“共享内容快照”解耦，建议新增：

```text
asset_document_snapshot_links
```

它的职责是：

```text
记录哪个 document 在哪次输入下引用了哪份 snapshot
```

建议字段如下：

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | 映射 ID | 系统生成 UUID / ULID |
| `document_id` | 哪个逻辑文档 | 关联 `asset_documents.id` |
| `document_snapshot_id` | 引用了哪份共享内容快照 | 关联 `asset_document_snapshots.id` |
| `source_batch_id` | 这次映射来自哪个输入批次 | 当前 `asset_source_batches.id` |
| `relative_path` | 本次输入中的相对路径 | 扫描输入得到 |
| `source_uri` | 本次输入中的原始位置 | 实际读取位置 |
| `title` | 该文档在本次映射下的标题 | 解析或文件名推断 |
| `scope_json` | 该文档在本次映射下的文档级范围 | 批次默认值 + 文档级规则 |
| `tags_json` | 该文档在本次映射下的文档级标签 | 批次标签 + 文档级标签 |
| `linked_at` | 建立映射时间 | 系统时间 |
| `metadata_json` | 扩展信息 | 系统补充 |

### 13.1.7 为什么路径和来源放在映射表里

因为这些信息本质上属于：

```text
某个 document 这次如何引用这份内容
```

而不是内容快照本身的共性属性。

例如：

- `relative_path`
- `source_uri`
- 文档级 `scope_json`
- 文档级 `tags_json`

如果把这些都放进共享 snapshot，本体就会被文档专属信息污染，破坏共享语义。

所以当前正式建议是：

| 信息类型 | 放在哪 |
|---|---|
| 共享内容属性 | `asset_document_snapshots` |
| 文档专属引用属性 | `asset_document_snapshot_links` |

### 13.1.8 对 Build 的影响

这次修订不会推翻 build 架构，但会更精确。

修订后的 build 语义应理解为：

```text
build 仍然按 document 粒度建立，
但选择对象是可共享的 snapshot。
```

所以 `asset_build_document_snapshots` 仍然合理，只是字段语义要按下面理解：

```text
document_id            = 哪个逻辑文档
document_snapshot_id   = 该文档在本次 build 采用哪份共享内容快照
```

因此即使：

- 文档 A
- 文档 C

共享同一份 snapshot，build 中仍然可以同时存在：

```text
document_A -> snapshot_S1
document_C -> snapshot_S1
```

这正是我们想要的效果：

- 内容只存一份
- 文档身份仍然保留两份
- build 仍然按文档视图成立

### 13.1.9 对下游 `raw_segments / relations / retrieval_units` 的影响

这次修订意味着：

- `asset_raw_segments`
- `asset_raw_segment_relations`
- `asset_retrieval_units`

都应理解为：

```text
挂在共享 snapshot 之下
```

而不是挂在“文档专属快照”之下。

这样才能做到：

- 相同内容只生成一套下游对象
- 多文档共享时不重复切片、不重复建关系、不重复建 retrieval unit

### 13.1.10 对 Mining 判断逻辑的影响

这次修订后，Mining 的判断要分成两层：

#### 层 1：判断逻辑文档身份

仍然用：

```text
document_key
```

去判断这是不是同一个逻辑文档。

#### 层 2：判断是否复用共享 snapshot

用：

```text
normalized_content_hash
```

去判断全库是否已有相同内容快照。

这意味着：

```text
document_key 用于“文档身份”
normalized_content_hash 用于“内容复用”
```

两者职责不同，不能混用。

### 13.1.11 对当前主架构的影响大小

这个修订不会推翻当前主架构。

不变的部分：

1. 三库边界不变
2. `build -> release` 主链不变
3. `mining_runtime` 的作用不变
4. `Serving` 只读 active release 的边界不变

变化的部分只有两点：

1. `asset_document_snapshots` 的语义，从“文档专属快照”修正为“共享内容快照”
2. 新增 `asset_document_snapshot_links`，把 document 身份和 snapshot 内容解耦

所以这是一次：

```text
语义修正 + 一张映射表补充
```

不是推翻式重构。

---

## 14. 最终结论

这套数据库架构的本质是：

```text
用 document snapshot 承接内容变化，
用 build 承接一次完整知识视图，
用 release / publish 承接正式对外生效。
```

它当前能做到：

1. 内容稳定存储
2. publish 有正式语义
3. Serving 查询边界清晰
4. 回退自然成立
5. 不重复复制大对象
6. 支持未来增量
7. 支持未来个性化扩展

如果后续继续往下审，这套方案最关键的三张表是：

- `asset_document_snapshots`
- `asset_build_document_snapshots`
- `asset_publish_releases`

因为这三张表决定了这套架构是否真正成立。
