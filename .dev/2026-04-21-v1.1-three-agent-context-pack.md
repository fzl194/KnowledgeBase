# CoreMasterKB 1.1 三方上下文包

- 日期：2026-04-21
- 作者：Codex
- 状态：待管理员审查
- 用途：给 `claude-mining`、`claude-serving`、`claude-llm` 三位执行人提供统一上下文、边界和目标态说明

---

## 1. 当前我们到底在做什么

CoreMasterKB 当前不是一个“文档搜索小工具”，也不是一个“只查命令”的定制系统。

它的目标是构建一套能被 Agent 作为 Skill 调用的知识后台。

这套后台需要满足：

1. 数据源复杂且持续变化  
   不只是产品文档，未来还会有专家文档、项目文档、PDF、HTML、TXT、DOCX 等。

2. 查询意图复杂且持续演进  
   不只是“某命令怎么写”，还包括概念解释、参数说明、步骤、注意事项、故障处理、适用范围、差异对比、上下文扩展等。

3. 系统要适合 Agent 调用  
   输出不能是“写死字段 + 定制问答逻辑”，而应该是可被上层 Agent 消化的上下文包和证据材料。

4. 短期内会有大量迭代  
   所以当前 1.1 的优先级不是“一次做完”，而是把主干方向定对，让后续扩展不会反复推翻。

一句话说：

```text
我们在做的是 Agent Knowledge Backend，
不是命令查询工具，也不是单轮 RAG demo。
```

---

## 2. 当前统一目标态

当前 1.1 的主干目标已经收敛为下面这条链路：

```text
原始资料
  -> Mining 解析与结构化
  -> asset_core 知识资产库
  -> Serving 检索与上下文扩展
  -> Agent Skill 消费

同时：
Mining / Serving 可以调用独立的 LLM Runtime 服务
```

### 2.1 核心原则

1. `asset_core` 是事实资产库  
   它保存当前 active 的知识资产，不保存复杂运行态噪音。

2. `mining_runtime` 是挖掘过程状态库  
   它服务于 run、断点续跑、阶段诊断、失败定位。

3. `agent_llm_runtime` 是独立 LLM 服务库  
   它服务于 prompt、task、attempt、result、event，不属于 Asset DB。

4. Serving 只读 active 资产  
   Serving 不参与生产资产、不改发布状态、不依赖 Mining 内部实现。

5. 当前检索主路径不再是 canonical  
   1.1 主路径应该围绕：

```text
raw_documents
  -> raw_segments
  -> raw_segment_relations
  -> retrieval_units
```

6. LLM 是增强器，不是事实源  
   无论在 Mining 还是 Serving，LLM 输出都不能替代原始事实。

## 3. 当前数据库边界

现在数据库目录已经重组为：

```text
databases/
  asset_core/
  mining_runtime/
  agent_llm_runtime/
```

### 3.1 asset_core

定位：

```text
Mining 写
Serving 读
只放当前 active 可服务知识资产
```

当前主要表：

- `asset_source_batches`
- `asset_publish_versions`
- `asset_batch_documents`
- `asset_raw_documents`
- `asset_raw_segments`
- `asset_raw_segment_relations`
- `asset_retrieval_units`
- `asset_retrieval_units_fts`
- `asset_retrieval_embeddings`（预留）
- `asset_canonical_segments`（保留但降级）
- `asset_canonical_segment_sources`（保留但降级）

### 3.2 mining_runtime

定位：

```text
只给 Mining 自己使用
记录过程，不记录最终资产事实
```

当前主要表：

- `mining_runs`
- `mining_run_documents`
- `mining_run_stage_events`

### 3.3 agent_llm_runtime

定位：

```text
独立 LLM Runtime 服务使用
Mining / Serving 通过 client / service 调用
```

当前主要表：

- `agent_llm_prompt_templates`
- `agent_llm_tasks`
- `agent_llm_requests`
- `agent_llm_attempts`
- `agent_llm_results`
- `agent_llm_events`

### 3.4 关于 Mining 涉及两个 DB 是否合并

当前结论已经明确：

```text
正式设计不合并
```

也就是：

- `asset_core` 和 `mining_runtime` 逻辑上必须分开
- 如果本地 dev 以后为了方便临时放进同一个 SQLite 文件，只能视为调试便利
- 不能把“物理同文件”误当成“正式架构合库”

原因：

1. `asset_core` 要稳定、可发布、可被 Serving 长期只读
2. `mining_runtime` 有大量中间态、失败态、重试态、诊断信息
3. 两者生命周期和清理策略完全不同

## 4. 三位执行人的共同上下文

这部分三个人都应该先知道。

### 4.1 当前不是做“最终正确实现”，而是做“正确主干”

我们当前更看重：

1. 边界对不对
2. 中间契约稳不稳
3. 后续扩展是否顺滑

我们当前不追求：

1. 一次把检索质量做到最终态
2. 一次把 LLM 接入做到最强
3. 一次把所有文件类型解析完
4. 一次把多版本管理做成完整 Git 模型

### 4.2 当前系统的演进方向

后续系统会继续演进到：

- Hybrid retrieval
- Embedding / vector search
- Rerank
- LLM query rewrite
- LLM retrieval planning
- Relation-based context expansion
- 进一步的图化关系
- 更复杂的数据源解析

所以三位执行人都必须避免：

```text
把自己的模块写成只能服务当前一个场景的定制实现
```

### 4.3 当前最重要的统一抽象

这几个抽象现在要尽量稳住：

1. `raw segment` 是事实单元
2. `retrieval unit` 是检索单元
3. `raw segment relation` 是上下文关系
4. `ContextPack / EvidencePack` 是 Serving 输出
5. `LLM Runtime Task` 是 LLM 调用抽象

## 5. 给 claude-mining 的完整上下文

## 5.1 他的角色

`claude-mining` 负责的是：

```text
把复杂原始资料变成结构化、可检索、可追溯的知识资产
```

他的价值不在于“把文本存进数据库”，而在于：

1. 把原始资料整理成稳定资产
2. 把篇章结构和来源定位保留下来
3. 把对检索有价值的结构化信息尽量抽出来
4. 为 Serving 提供通用而不是定制化的检索基础

## 5.2 他面对的问题本质

原始资料未来不会稳定：

- 可能是 Markdown
- 可能是 TXT
- 可能是 HTML
- 可能是 PDF / DOCX
- 可能是专家文档
- 可能没有产品/网元这些字段

所以 Mining 的目标不是：

```text
把某一种文档喂给某一种规则，然后满足某一种查询
```

而是：

```text
把“文档 -> 结构 -> 片段 -> 关系 -> 检索单元”这条通用生产链做好
```

## 5.3 当前他应该对齐的目标态

### A. 原始事实层

把每个文件登记成 `asset_raw_documents`，把可解析内容落到 `asset_raw_segments`。

关键要求：

- 文档可回溯
- 片段可回溯
- section_path 要稳定
- structure_json 要尽量保真
- source_offsets_json 要可定位

### B. 上下文关系层

生成 `asset_raw_segment_relations`。

这是当前 Mining 必须重视的重点，不是可有可无的增强。

最少要稳定生成：

- `previous`
- `next`
- `same_section`
- `same_parent_section`
- `section_header_of`

### C. 检索单元层

生成 `asset_retrieval_units`。

这里最关键的思想是：

```text
retrieval_unit 不是去重结果
retrieval_unit 是面向检索的封装视图
```

也就是说：

- raw segment 是事实源
- retrieval unit 是检索入口

同一个 raw segment 可以生成多个 retrieval unit：

- `raw_text`
- `contextual_text`
- `summary`
- `generated_question`
- `table_row`
- `entity_card`

第一版不要求全做完，但设计必须往这个方向走。

## 5.4 他不该做什么

1. 不要把 Serving 当前某个查询逻辑写死到 schema 中
2. 不要要求未来语料必须带 manifest 或固定元数据文件
3. 不要把 canonical 当 1.1 主路径继续强化
4. 不要把 LLM 结果当原始事实
5. 不要为了某个命令场景造一堆命令专用外层列

## 5.5 他需要和其他人怎样对接

### 对 Serving

他提供的是：

- active `asset_core`
- 稳定的 `retrieval_units`
- 可下钻的 `raw_segments`
- 可扩展的 `raw_segment_relations`

他不应该依赖 Serving 的具体 SQL 写法。

### 对 LLM

他应该把 LLM 看成一个“可选增强器”。

可以用在：

- generated question 生成
- summary 生成
- 语义角色候选增强
- 实体候选增强

但所有这类输出都必须：

- 可追溯
- 弱引用到 LLM runtime
- 不能覆盖原始事实

## 5.6 给他的评价标准

不是“是不是把文档都处理完了”，而是：

1. 资产是不是稳定
2. 结构是不是保住了
3. 来源是不是可回溯
4. 关系是不是足够支撑上下文扩展
5. retrieval unit 抽象是不是走对了

## 6. 给 claude-serving 的完整上下文

## 6.1 他的角色

`claude-serving` 负责的是：

```text
把知识资产变成 Agent 能消费的上下文服务
```

他不负责：

- 解析文档
- 生产资产
- 发布版本
- 重写数据库事实

## 6.2 他面对的问题本质

Serving 不是做一个“固定问答接口”，而是在做：

```text
Agent Skill 的知识调用后端
```

这意味着：

1. 查询意图会越来越复杂
2. 上层 Agent 可能会传 scope、entity、intent，也可能什么都不传
3. 后续完全可能把部分理解和规划交给 LLM
4. 输出不能被某个单场景绑死

## 6.3 当前他应该对齐的目标态

### A. 以 retrieval_units 为主检索入口

当前 1.1 主路径不应该再围绕 canonical。

主检索入口应该是：

```text
asset_retrieval_units
```

然后：

```text
retrieval_units
  -> source_refs_json
  -> raw_segments
  -> raw_documents
  -> raw_segment_relations
```

### B. 输出通用 ContextPack

Serving 返回的东西要适合 Agent 消费，而不是适合某一个前端页面。

返回结构应该是通用的 Evidence / Context 包。

核心模块应包括：

- query understanding 结果
- items / evidence
- sources
- relations
- variants
- conflicts
- gaps
- suggestions
- debug（可选）

### C. 设计成可插入 LLM 的查询链

Serving 当前不能强依赖 LLM，但要预留下面这些插点：

- query rewrite
- intent extraction
- entity enrichment
- retrieval planning
- rerank
- context compression

也就是说当前就要有：

```text
query understanding
-> query plan
-> retrieval
-> expansion
-> assembly
```

即便第一版 planner 很简单，也不能没有这个抽象。

## 6.4 他不该做什么

1. 不要继续把 Serving 写成 command-only 系统
2. 不要强依赖 JSON 里必须存在某个子字段才能检索
3. 不要把 Mining 的当前实现细节当成长期契约
4. 不要把查询逻辑固化成一堆 endpoint 特判
5. 不要把最终答案生成混进 Serving 主职责

## 6.5 他需要和其他人怎样对接

### 对 Mining

他必须把 Mining 输出看成：

```text
尽力结构化，但不完美
```

所以 Serving 读取时必须：

- 允许字段缺失
- 允许 JSON 形态不完美
- 保留 fallback 路径

### 对 LLM

他应该把 LLM 看成在线增强器，而不是基础依赖。

原则：

- 有 LLM 时增强理解和排序
- 没有 LLM 时基础检索仍然可用

## 6.6 给他的评价标准

不是“某条命令能不能查到”，而是：

1. 查询链路抽象是否通用
2. 对数据契约是否足够宽容
3. 输出是否适合 Agent 使用
4. 是否已经把 LLM 插点留好
5. 是否走向 hybrid retrieval，而不是继续写死规则问答

## 7. 给 claude-llm 的完整上下文

## 7.1 他的角色

`claude-llm` 不是来写业务逻辑的。

他负责的是：

```text
把 LLM 能力抽象成一个独立、稳定、可追踪、可审计的 runtime 服务
```

他的目标不是回答业务问题，而是给 Mining 和 Serving 提供统一 LLM 能力底座。

## 7.2 他面对的问题本质

如果 LLM 不独立抽象，后面一定会出现：

- Mining 自己写一套 prompt / retry / parse
- Serving 再写一套 prompt / retry / parse
- 两边日志、幂等、模板、schema 验证全分裂

所以他要解决的是：

```text
LLM 调用能力的平台化
```

## 7.3 当前他应该对齐的目标态

### A. 统一任务模型

核心抽象是：

- prompt template
- logical task
- request
- attempt
- result
- event

这几层必须明确分开。

### B. 同时服务 Mining 和 Serving

他不是只给 Serving 用，也不是只给 Mining 用。

两边都要能调用。

典型场景：

#### Mining 侧

- summary generation
- generated question generation
- semantic enrichment
- entity enrichment

#### Serving 侧

- query rewrite
- intent extraction
- retrieval planning
- rerank
- context compression

### C. 强调幂等、追溯、审计

因为后面只要 LLM 接入变深，就一定会遇到：

- 失败重试
- provider 波动
- JSON 解析失败
- schema 不匹配
- 重复提交
- 成本统计

所以 runtime 设计里必须把这些前置考虑进去。

## 7.4 他不该做什么

1. 不要把业务逻辑耦合进 runtime
2. 不要直接写 asset_core 表
3. 不要假设只有一个 provider 或一个模型
4. 不要只做“调用一下 API”的轻封装
5. 不要把 prompt 版本和结果追溯省掉

## 7.5 他需要和其他人怎样对接

### 对 Mining / Serving

他提供的是：

- 一个 runtime 服务
- 一套任务表
- 一个清晰 client 接口
- 一套输出 schema 验证机制

他不负责决定：

- 哪个业务点一定要调用 LLM
- LLM 结果如何写回 asset_core
- Serving 的检索排序策略

## 7.6 给他的评价标准

不是“能不能调通模型”，而是：

1. runtime 抽象是否完整
2. 是否适合两边复用
3. 是否有足够的追溯和幂等能力
4. 是否能承接后续 provider / template / schema 演进

## 8. 三个人之间真正的接口

## 8.1 Mining -> Serving

接口不是 Python 代码，不是函数调用，而是：

```text
asset_core 数据契约
```

主要是：

- `raw_documents`
- `raw_segments`
- `raw_segment_relations`
- `retrieval_units`
- `publish_versions`

## 8.2 Mining / Serving -> LLM

接口不是共享表写入，而是：

```text
LLM Runtime client / service
```

返回结果后，业务侧只保留弱引用。

## 8.3 管理员真正要控制的东西

管理员后续分任务时，最重要的不是把代码切太细，而是把下面四件事切清楚：

1. 每个人负责的数据库边界
2. 每个人负责的抽象层
3. 每个人不应该跨过去做的事
4. 每个人产出后谁消费、怎么验证

---

## 9. 当前建议的发任务口径

如果后面正式发任务，我建议三个任务的口径不要写成“改几个文件”，而要写成下面这种目标态导向。

### 给 Mining 的口径

```text
围绕 raw -> relations -> retrieval_units 这条主链，
把离线知识资产生产路径拉到 1.1 正轨。
```

### 给 Serving 的口径

```text
围绕 retrieval_units + relations + ContextPack，
把在线知识服务重写成 Agent 可消费的通用检索后台。
```

### 给 LLM 的口径

```text
围绕 prompt / task / request / attempt / result / event，
做一个可被 Mining 和 Serving 共同调用的独立 LLM Runtime。
```

## 10. 当前最重要的收束结论

最后把目前最关键的结论再压缩一次，供后面发任务直接引用。

### 结论 1

当前系统是：

```text
Agent Knowledge Backend
```

不是命令查询工具。

### 结论 2

当前数据库正式边界是三套：

```text
asset_core
mining_runtime
agent_llm_runtime
```

### 结论 3

Mining 涉及的两个 DB：

```text
asset_core + mining_runtime
```

正式设计不合并。

### 结论 4

Serving 1.1 主路径应从 canonical 切到：

```text
retrieval_units + raw_segments + raw_segment_relations
```

### 结论 5

LLM Runtime 必须独立，不让 Mining / Serving 各自私建一套 LLM 调用体系。
