# Knowledge Mining v1.1 Pipeline 演进方案

- 日期：2026-04-21
- 作者：claude-mining
- 状态：待审核

## 背景

v0.5 pipeline：`ingest → profile → parse → segment → canonicalize → publish`

v1.1 pipeline：`ingest → profile → parse → segment → build_relations → build_retrieval_units → publish`

核心变化：
- **删除 canonical**：不再做三层去重，直接以 raw_segment 为检索单元
- **新增 relations**：segment 间 previous/next/same_section/section_header_of 关系
- **新增 retrieval_units**：每个 segment 生成一个 contextual_text 类型的检索单元，拼接标题+实体+原文
- **增量挖掘**：content_hash 对比实现 NEW/UPDATE/SKIP 分类
- **Mining Runtime**：独立 SQLite 记录挖掘运行状态，支持断点续跑

## Schema 确认

v1.1 schema 由 Codex 构建完成：`databases/asset_core/schemas/001_asset_core.sqlite.sql`

已确认符合演进要求：
- `asset_raw_documents` / `asset_raw_segments` 无 `publish_version_id`
- `asset_raw_segment_relations` 支持 10 种 relation_type
- `asset_retrieval_units` 含 FTS 触发器
- `asset_batch_documents` 支持多对多挂载
- `asset_publish_versions` 保留作为发布控制边界（`one_active` 唯一索引）

## T3: 新建 relations 构建模块

### 新文件

`knowledge_mining/mining/relations.py`

### 输入/输出

```python
@dataclass(frozen=True)
class SegmentRelation:
    source_segment_key: str  # "doc_key#index"
    target_segment_key: str
    relation_type: str       # previous/next/same_section/section_header_of
    weight: float = 1.0
    confidence: float = 1.0
    distance: int = 0
    metadata_json: dict = field(default_factory=dict)

def build_relations(segments: list[RawSegmentData]) -> list[SegmentRelation]: ...
```

relation 的 key 使用 `{document_key}#{segment_index}` 格式，publishing 阶段替换为真实 ID。

### 构建规则

**previous / next**：按 segment_index 排序，相邻 segments 互连，distance=1

**same_section**：section_path 相同的 segments 两两建立关系，distance=|i-j|。同 section 超过 20 个 segment 时只连接相邻的（避免 N²）

**section_header_of**：heading segment 与其子路径下第一个 segment 建立

**same_parent_section**：v1.1 暂不实现

### 与 schema 对齐

`relation_type` 在 CHECK 约束内：`previous, next, same_section, same_parent_section, section_header_of, references, elaborates, condition, contrast, other`

T3 只生成前 5 种中的 4 种（`same_parent_section` 暂不生成）。

### 验证

新建 `tests/test_relations.py`：previous/next 链、same_section 分组、section_header_of、空/单 segment 边界

### 输入/输出

```python
@dataclass(frozen=True)
class RetrievalUnitData:
    unit_key: str          # "ru-{segment_key}"
    unit_type: str         # "contextual_text"
    target_type: str       # "raw_segment"

    segments: list[RawSegmentData],
```


```python
    """CJK 字符间插入空格，让 unicode61 按单字索引。"""
```

## T5: 重写 publishing

整个 `publishing/__init__.py` 重写。

### 函数签名

```python
def publish(
    documents: list[RawDocumentData],
    segments: list[RawSegmentData],
    relations: list[SegmentRelation],
    retrieval_units: list[RetrievalUnitData],
    db_path: Path,
    batch_params: BatchParams | None = None,
) -> dict[str, Any]: ...
```

### 新流程

1. 创建 source_batch + staging publish_version
2. 写 raw_documents（无 publish_version_id）
3. 写 batch_documents（挂载批次-文档关系）
4. 写 raw_segments（无 publish_version_id）
5. 写 raw_segment_relations（key → ID 映射）
6. 写 retrieval_units（key → ID 映射）
7. 验证：raw_documents > 0, raw_segments > 0, retrieval_units > 0
8. 原子激活：archive old active → activate new staging
9. 失败回滚

### 关键设计

**ID 映射**：relations 和 retrieval_units 中的 segment_key/document_key 需映射为数据库 UUID

**验证规则**：只检查 raw_documents/raw_segments/retrieval_units > 0，不再检查 canonical

**激活逻辑**：保留现有 `activate_version` 原子切换

### 验证

`test_publishing.py` 全面重写

## T6: 重写 run.py pipeline

### 新流程

```python
# Step 1: Ingest
docs, ingest_summary = ingest_directory(input_path, batch_params)

# Step 2: Profile
profiles = [build_profile(d) for d in docs]

# Step 3: Parse + Segment
for doc, profile in zip(docs, profiles):
    ...  # parse → segment

# Step 4: Build relations（按文档分组）
for doc_key, group in groupby(all_segments, key=lambda s: s.document_key):
    all_relations.extend(build_relations(list(group)))

# Step 5: Build retrieval units
all_retrieval_units = build_retrieval_units(all_segments, all_relations)

# Step 6: Publish
pub_result = publish(
    documents=docs,
    segments=all_segments,
    relations=all_relations,
    retrieval_units=all_retrieval_units,
    db_path=db_path,
    batch_params=batch_params,
)
```

### 函数签名

去掉 `segment_enricher` 参数，新增 `chunk_size`/`chunk_overlap`。

### 返回值

包含 raw_segments、relations、retrieval_units 计数，以及 active_version_id、status、version_code。

### 验证

`test_pipeline.py` 确认 end-to-end：md 输入 → db 有 relations 和 retrieval_units

## T8: Mining Runtime 独立库

### 新文件

`knowledge_mining/mining/runtime.py`

### 数据库文件

`data/<corpus_name>/mining_runtime.sqlite`（独立于 asset.sqlite）

### 表结构

**mining_runs**：id, source_batch_id, input_path, status(running/completed/interrupted/failed), 统计字段, 时间戳, metadata_json

**mining_run_documents**：id, run_id, document_key, content_hash, action(NEW/UPDATE/SKIP), status(pending/processing/committed/failed), existing_document_id, 时间戳, metadata_json

**mining_run_stage_events**：id, run_document_id, stage(parse/segment/build_relations/build_retrieval_units/commit), status(started/completed/failed), duration_ms, output_summary, error_message

### 与 pipeline 集成

run_pipeline() 叠加：
1. 创建 mining_run (status=running)
2. 扫描目录，对比 Asset DB 中的 content_hash
3. 每个文档写入 mining_run_document (action=NEW/UPDATE/SKIP)
4. 只处理 NEW/UPDATE，每个阶段记录 stage_event
5. 更新统计，标记 completed

### 断点续跑

`resume_pipeline()` 找出 pending/processing/failed 的文档，继续处理。

### 验证

新建 `tests/test_runtime.py`：状态机、stage_events、content_hash 分类
