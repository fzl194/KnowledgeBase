# CoreMasterKB 1.1 数据库最终版提案

- 日期：2026-04-21
- 作者：Codex
- 状态：待评审
- 目标：给出当前阶段可落地、可扩展、不过度复杂的最终数据库方案。

## 1. 设计结论

本版明确放弃以下两条路线：

1. 放弃“每个 publish version 物理全量复制一套资产表”
2. 放弃 `document_revision` / Git 式增量版本模型

本版采用的最终思路是：

```text
1. Asset 表只存当前 active 知识资产
2. publish_version 保留，但只做“发布控制边界”，不挂到每张资产表
3. Mining Runtime 单独存处理过程状态
4. LLM Runtime 单独存 LLM 调用过程
5. Serving 永远只读当前 active 资产
6. 不重复持久化存储历史全量数据
```

一句话概括：

```text
当前态资产 + 最小发布控制 + 独立运行态状态库
```

## 2. 为什么这样定

当前我们的目标不是做完整版本管理系统，而是满足：

1. 数据不重复存储
2. Serving 查询简单
3. 有最低限度的发布边界
4. 有最低限度的回退与追溯能力

如果走“每表挂版本”：

- 设计复杂
- 字段重复
- 查询复杂

如果走“document revision / Git”：

- 查询链路变长
- 版本语义过重
- 当前阶段过度设计

如果彻底删掉 `publish_versions`：

- Mining 和 Serving 之间没有稳定发布边界
- 半成品资产可能暴露给 Serving
- 没有最基本的切换和回退点

所以最终取中间方案：

```text
publish_version 保留
但只表达“某次构建是否成为当前可服务版本”
不做历史视图系统
```

## 3. 三库边界

## 3.1 Knowledge Asset DB

用途：

- 存当前 active 知识资产
- 给 Serving 查询
- 不存复杂运行态状态

## 3.2 Mining Runtime DB

用途：

- 跟踪一次挖掘执行的过程状态
- 支持断点续跑
- 支持阶段诊断

## 3.3 LLM Runtime DB

用途：

- 由 `claude-llm` 独立维护
- 存储 LLM 任务、prompt、attempt、解析结果、队列事件

## 4. 总体结构

```text
source_batch
  -> mining_run
  -> staging asset build
  -> publish_version
  -> active asset tables

Serving
  -> read active asset tables only
```

关键点：

- `publish_version` 不和 `raw_documents/raw_segments/retrieval_units` 做外键绑定
- 资产表不带 `publish_version_id`
- 资产表也不带 `start/end version`
- 当前版本通过“发布切换”形成，而不是通过“按版本过滤大表”形成

## 5. 发布机制

本版的核心不是历史版本重放，而是：

```text
Mining 在 staging 区构建
校验通过后替换 active 资产
Serving 始终只读 active 资产
```

### 5.1 staging 形式

可以有两种实现，逻辑等价：

#### 方案 A：单独 staging SQLite / staging schema

- dev：单独 `asset_staging.sqlite`
- prod：单独 `asset_staging.*` schema 或独立 staging 库

#### 方案 B：同库内 staging 表组

- `asset_staging_raw_documents`
- `asset_staging_raw_segments`
- `asset_staging_raw_segment_relations`
- `asset_staging_retrieval_units`

当前阶段更推荐：

```text
dev 用独立 staging sqlite
prod 再决定是 staging schema 还是 staging db
```

### 5.2 发布切换流程

```text
1. 创建 source_batch
2. 创建 mining_run
3. 在 staging 资产区完成 parse/segment/relation/retrieval_unit 构建
4. 校验通过后：
   - old active publish_version -> archived
   - new publish_version -> active
   - staging 资产替换 active 资产
   - 重建 FTS / 检索索引
5. Serving 继续只读 active 资产
```

### 5.3 rollback 能力

本版不支持任意历史版本精确重放，但保留最低限度回退能力：

1. `asset_publish_versions` 记录历史发布元信息
2. 每次发布可选保存一份发布包路径到 `metadata_json`
3. 出问题时可以回退到上一份 active 资产包

所以这里的回退不是“在线按版本查询”，而是“工程级回切到上一个发布包”。

## Part 1. 库级边界与总流转

这一部分先回答最上层的问题：整个数据库架构到底长什么样，以及表和表之间不是静态摆在那里，而是如何沿着一条链路流动。

### 1.1 三库各自的职责

#### `asset_core`

这是正式知识资产库。它回答的是：

```text
系统里有哪些知识内容
一次知识构建 build 选择了哪些内容
哪个 release 当前正式对外生效
```

Serving 只读这个库。

#### `mining_runtime`

这是挖掘运行态库。它回答的是：

```text
这次挖掘 run 到哪一步了
哪个文档失败了
哪个阶段失败了
这次 run 最终是否构建出了可发布 build
```

它不存正式知识资产。

#### `agent_llm_runtime`

这是独立 LLM 服务库。它回答的是：

```text
谁提交了 LLM 任务
用了哪个 prompt 模板
发了哪些 request
尝试了几次
最终产出了什么结构化结果
```

它也不存正式知识资产。

### 1.2 为什么必须拆成三库

因为这是三类完全不同的数据：

| 类别 | 代表内容 | 生命周期 | 读写模式 |
|---|---|---|---|
| 正式知识资产 | documents / snapshots / segments / relations / retrieval_units / builds / releases | 长生命周期，需要稳定回溯 | Mining 写入，Serving 高频读取 |
| 挖掘运行态 | run / per-document status / stage events | 短中生命周期，允许失败、重试、清理 | Mining 高频写入 |
| LLM 运行态 | task / request / attempt / result / event | 独立演进，可能跨 Mining 与 Serving 复用 | LLM service 主写，业务侧只引用 |

如果把这三类东西混在一个库里，会立即出现三个问题：

1. Serving 会被迫看到大量挖掘中间态和失败态。
2. 资产表会被运行态状态污染，字段语义开始混乱。
3. LLM 任务会和知识内容本体耦合，后续独立扩展困难。

所以当前库级架构必须是：

```text
asset_core         = 正式知识资产
mining_runtime     = 挖掘过程状态
agent_llm_runtime  = 独立 LLM 运行态
```

### 1.3 从输入到查询的总数据流

整个系统的数据流应该这样理解：

```text
输入资料
  -> source batch
  -> mining run
  -> document snapshot
  -> raw segments
  -> raw segment relations
  -> retrieval units
  -> build
  -> release
  -> serving query
```

这条链的每一段负责的问题不同：

| 阶段 | 作用 |
|---|---|
| `source batch` | 给输入资料一个批次身份 |
| `mining run` | 记录这次处理过程 |
| `document snapshot` | 固化某个逻辑文档的一份不可变内容快照 |
| `raw segments` | 沉淀原始事实片段 |
| `relations` | 沉淀篇章关系与上下文关系 |
| `retrieval units` | 提供可检索、可返回给 Agent 的检索单元 |
| `build` | 选择这次知识视图采用哪些 document snapshot |
| `release` | 指定哪个 build 当前正式对外生效 |
| `serving query` | 只在 active release 对应的 build 范围内检索 |

### 1.4 三库之间如何交互

不是三库彼此强耦合外键，而是按职责单向流动：

#### `mining_runtime` -> `asset_core`

Mining 跑起来以后：

1. 在 `mining_runtime` 里登记 `run`
2. 记录每个文档当前状态
3. 文档处理成功后，把正式内容写入 `asset_core`
4. 当 build 构建完成并校验通过后，在 `asset_core` 中创建 release

也就是说，`mining_runtime` 负责“过程跟踪”，`asset_core` 负责“结果落地”。

#### `asset_core` -> `agent_serving`

Serving 不关心 run，也不关心 LLM task。Serving 只关心两件事：

1. 当前 active release 是谁
2. 这个 active release 指向哪个 build，以及这个 build 采用哪些 document snapshot

然后它只在这个范围内检索 `asset_retrieval_units`。

#### `asset_core` <-> `agent_llm_runtime`

这里不是强外键，而是弱引用：

- Mining 可能调用 LLM 来生成 summary / question / enrichment
- Serving 可能调用 LLM 来做 query rewrite / rerank / context compression

但资产库中只保留：

```text
llm_result_refs_json
```

用于回溯“这条增强内容来自哪个 LLM task/result”，而不会把 LLM 运行态表直接塞进资产库。

### 1.5 当前架构已经能保证什么

仅从 Part 1 这层看，这套架构已经能保证：

1. 正式知识资产和挖掘过程状态分层。
2. publish 有正式承载对象，不再只是补救动作。
3. Serving 查询边界明确，只读 active release 对应的 build。
4. LLM 可以作为增强能力接入，但不会污染资产主库。
5. 后续个性化知识库如果要做，也是在 `build/release` 之上叠规则层，而不是推翻底层资产结构。

这一部分只把库级边界和总流转钉住，还没有展开：

1. `asset_documents` / `asset_document_snapshots` 的字段细节

## Part 2. 公共内容身份层

这一部分只回答三个问题：

1. 一批输入资料如何进入系统。
2. 什么叫“逻辑文档”。
3. 什么叫“文档快照”，以及为什么后续所有正式内容都应该挂在快照之下。

涉及三张表：

- `asset_source_batches`
- `asset_documents`
- `asset_document_snapshots`

这一层的定位是：

```text
先把“输入是谁、文档是谁、文档这次的内容是什么”说清楚，
后面的 segment / relation / retrieval unit 才有稳定锚点。
```

### 2.1 `asset_source_batches`

#### 2.1.1 这张表解决什么问题

它解决的是：

```text
这次喂给系统的输入资料，是哪一批
```

注意这里记录的是“输入批次身份”，不是“运行过程状态”。

例如：

- 用户上传了一个文件夹
- 后端导入了一个 zip
- 管理员指定一个目录作为本次语料输入

这些都应该先形成一个 `source_batch`。

#### 2.1.2 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | 批次 ID | 系统生成 UUID / ULID |
| `batch_code` | 便于人读的批次编码 | 创建批次时按规则生成 |
| `source_type` | 输入来源类型，如 `folder_upload/folder_scan/api_import/manual_seed` | 由触发入口给出 |
| `description` | 对本批次的简短说明 | 调用方填写，可为空 |
| `created_by` | 谁创建的 | 用户、管理员或系统 |
| `created_at` | 创建时间 | 系统时间 |
| `metadata_json` | 扩展元信息 | 由调用方和系统共同补充 |

#### 2.1.3 `metadata_json` 里应该放什么

这一层可以放“批次级默认信息”，但不建议把大量业务字段拍成外层列。

推荐放：

```json
{
  "storage_root_uri": "file:///kb/input/2026-04-21/",
  "default_document_type": "command",
  "batch_scope": {
    "product": "CloudCore",
    "version": "V100R023C10"
  },
  "tags": ["coldstart", "commands"],
  "notes": "管理员导入的测试批次"
}
```

这里的意思是：

- 这批资料整体可能带一些默认范围和标签
- 但这些只是默认值
- 后续具体文档和具体 retrieval unit 可以覆盖、细化、补充

#### 2.1.4 这张表往下怎么流

它会流向两处：

1. `mining_runtime`
   - 让这次 `mining_run` 知道自己在处理哪一批输入
2. `asset_document_snapshots`
   - 让每份文档快照知道自己最初来自哪个批次

所以关系是：

```text
source_batch
  -> mining_run
  -> document_snapshot
```

---

### 2.2 `asset_documents`

#### 2.2.1 这张表解决什么问题

这张表不是文档内容表，而是“逻辑文档身份表”。

它回答的是：

```text
系统认为这是同一篇文档，还是另一篇文档
```

例如：

- `02_commands/05_add_apn.md`
- 后续它内容改了
- 或从 html 重新转成 md

只要我们认为它还是同一个逻辑文档，它就应该共享同一个 `document_id`。

#### 2.2.2 为什么要单独有这张表

因为如果没有这层，系统会把“文档身份”和“文档内容快照”混在一起。

后果是：

1. 你无法明确说“同一逻辑文档在不同时间有多个内容快照”
2. build 无法在一个稳定文档身份上选择采用哪个 snapshot
3. 后续做增量、回退、个性化视图时都会变乱

所以必须把两件事拆开：

```text
asset_documents           = 文档是谁
asset_document_snapshots  = 文档这次长什么样
```

#### 2.2.3 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | 逻辑文档 ID | 系统生成 UUID / ULID |
| `document_key` | 稳定文档键 | 由输入根目录相对路径、外部业务键或约定规则生成 |
| `document_name` | 文档展示名 | 优先来自标题，否则来自文件名 |
| `document_type` | 文档类型 | 批次默认值、目录规则、人工指定或后续抽取 |
| `metadata_json` | 稳定级扩展信息 | 系统与业务侧补充 |
| `created_at` | 首次创建时间 | 系统时间 |

#### 2.2.4 `document_key` 怎么定

这是这一层最关键的字段。

当前建议是：

- 对于“给定文件夹递归导入”的 M1 / 1.1 场景，优先使用：
  ```text
  相对输入根目录的路径
  ```
- 如果未来有更稳定的外部业务键，可以替换为：
  ```text
  外部业务键 / 内容系统 ID / 约定文档主键
  ```

也就是说，`document_key` 的职责是：

```text
在系统里稳定标识“这还是不是同一个逻辑文档”
```

不是用来表达文档内容版本。

#### 2.2.5 `document_name` 和 `document_type` 怎么来

##### `document_name`

优先级建议：

1. 文档解析后识别到的主标题
2. 文件名去后缀
3. 调用方显式指定名称

##### `document_type`

优先级建议：

1. 调用方明确指定
2. `source_batch.metadata_json.default_document_type`
3. 目录规则判断
4. 解析或分类器推断
5. `unknown`

注意：

`document_type` 是业务归类信号，不是路径事实。它可以随着规则升级而更准确，但 `document_key` 不应因此变化。

#### 2.2.6 这张表往下怎么流

它只往下流到一张表：

```text
asset_document_snapshots
```

也就是：

```text
一个逻辑文档
  -> 多个内容快照
```

---

### 2.3 `asset_document_snapshots`

#### 2.3.1 这张表解决什么问题

这是整个数据库方案的第一个核心表。

它回答的是：

```text
某个逻辑文档，在某次输入下，对应的是哪一份不可变内容快照
```

这张表一旦成立，后面的 segment / relation / retrieval unit 就都不需要直接挂版本号了。

#### 2.3.2 为什么“快照”是必要层

因为我们要同时满足：

1. 内容不重复全量复制
2. build / publish 有正式承载对象
3. 不给每条 segment / retrieval unit 挂版本字段
4. 支持未来增量
5. 支持未来回退与多视图

只有把变化收敛在“文档内容快照”这一层，上面 5 条才能同时成立。

如果没有快照层，版本变化只能落在两种地方：

1. 落在每张主表的每一行上  
   复杂度太高
2. 落在整库替换上  
   publish 没有正式数据库意义

所以 `document_snapshot` 是当前最小且必要的设计。

#### 2.3.3 建议字段

| 字段 | 含义 | 字段怎么来 |
|---|---|---|
| `id` | 文档快照 ID | 系统生成 UUID / ULID |
| `document_id` | 属于哪个逻辑文档 | 由 `asset_documents.id` 关联 |
| `source_batch_id` | 来自哪个输入批次 | 由当前 `source_batch` 填入 |
| `relative_path` | 相对输入根目录路径 | 扫描文件时得到 |
| `source_uri` | 原始来源位置 | 输入文件真实位置或对象存储 URI |
| `mime_type` | 输入格式，如 `text/markdown`、`text/plain`、`text/html` | 由扩展名、探测器或上游传入 |
| `content_hash` | 原始内容 hash | 对原始输入内容求 hash |
| `title` | 该快照的标题 | 解析结果或文件名推断 |
| `scope_json` | 文档级适用范围 | 批次默认值 + 文档级提取/覆盖 |
| `tags_json` | 文档级标签 | 批次默认标签 + 文档级标签 |
| `parser_profile_json` | 解析链信息 | 由 parser / converter / profile 写入 |
| `metadata_json` | 其他扩展信息 | 由系统补充 |
| `created_at` | 快照创建时间 | 系统时间 |

#### 2.3.4 每个字段具体怎么来

##### `document_id`

流程是：

1. 扫描输入文件
2. 按规则计算 `document_key`
3. 查 `asset_documents`
4. 如果存在，则复用旧 `document_id`
5. 如果不存在，则创建新的逻辑文档
6. 再创建新的 snapshot，挂到这个 `document_id` 下

##### `source_batch_id`

直接来自这次触发输入时创建的 `asset_source_batches.id`。

##### `relative_path`

由扫描输入目录时得到：

```text
文件相对本次输入根目录的稳定路径
```

它是当前 M1 / 1.1 阶段最实际的文档定位字段。

##### `source_uri`

这是系统真实读取位置。

例如：

- 本地绝对路径
- 文件存储 URI
- 对象存储地址

`relative_path` 用于“文档身份和展示”，`source_uri` 用于“真实回溯到原始位置”。

##### `mime_type`

优先级建议：

1. 上游显式传入
2. 按扩展名推断
3. 内容探测器判断
4. 默认 `application/octet-stream`

##### `content_hash`

对“原始输入内容”求 hash。

这里要注意：

- 不是对解析后文本求 hash
- 也不是对 segment 求 hash

它的职责是：

```text
判断这份输入内容是否和已有快照相同
```

##### `title`

优先级建议：

1. 解析出的主标题
2. 文档 front matter / 上游元信息
3. 文件名去后缀

##### `scope_json`

这不是固定外层列，因为未来范围维度会扩展。

当前来源应是：

1. `source_batch.metadata_json.batch_scope`
2. 文档级规则或目录推断
3. 文档解析或后续抽取补充

例如：

```json
{
  "product": "CloudCore",
  "version": "V100R023C10",
  "network_element": "PGW-C"
}
```

##### `tags_json`

来源与 `scope_json` 类似：

1. 批次默认标签
2. 文档路径或目录规则标签
3. 文档解析或后续抽取标签

##### `parser_profile_json`

这一列是为了强回溯“这份快照是怎么来的”。

建议记录：

```json
{
  "parser": "markdown_parser_v1",
  "converter": "productdoc_to_md_v3",
  "input_format": "html",
  "normalized_format": "markdown",
  "profile": "md_m1"
}
```

也就是说，它不是业务知识，而是加工链路元信息。

#### 2.3.5 快照怎么和内容去重结合

当前建议的规则是：

```text
同一个 document_id 下，
如果新输入内容的 content_hash 和已有 snapshot 相同，
则直接复用旧 snapshot；
如果 content_hash 不同，
则创建新 snapshot。
```

这样做到的是：

1. 文档没变，不重复生成快照
2. 文档变了，只新增这一篇的快照
3. 不需要重新复制整个知识库

#### 2.3.6 这张表往下怎么流

这是公共内容身份层和正式内容层之间的桥。

一个 `document_snapshot` 会往下派生三类对象：

```text
document_snapshot
  -> raw_segments
  -> raw_segment_relations
  -> retrieval_units
```

也就是说，后面所有正式知识内容都应该挂在 `document_snapshot_id` 下，而不是直接挂 `document_id`，也不是直接挂 `publish/release`。

#### 2.3.7 Part 2 的小结

到这里，公共内容身份层已经成立：

```text
source_batch

  -> document_snapshot
```

它们分别负责：

只要这一层成立，后面：

- segment
- relation
- retrieval_unit
- build
- release

就都有了稳定的身份锚点。
| `processing_profile_json` | 解析与抽取过程信息 |

## 7. 增量处理逻辑

## 7.1 发现阶段

对输入目录递归扫描。

对每个文件：

1. 计算 `document_key = relative_path`
2. 计算 `content_hash`
3. 查询当前 active `asset_raw_documents`

分类：

- 不存在：`NEW`
- 存在且 hash 相同：`SKIP`
- 存在但 hash 不同：`UPDATE`

## 7.2 处理阶段

只处理 `NEW / UPDATE`。

每个文档完整完成：

```text
parse
-> segment
-> enrich
-> build_relations
-> build_retrieval_units
```

全部在 staging 资产区完成。

## 7.3 发布阶段

当本次 run 完成并通过校验后：

1. old active `publish_version` -> `archived`
2. new `publish_version` -> `active`
3. staging 资产替换 active 资产
4. 重建 FTS / embedding 索引

Serving 在任何时候只读取 active 资产。

## 8. 断点续跑

续跑依据是 Mining Runtime，而不是 Asset 表。

逻辑：

- `committed` 的文档跳过
- `pending` 的文档继续做
- `processing` 的文档按未完成处理重新跑
- `failed` 的文档按策略重试

所以：

```text
资产表不承担阶段状态职责
```

## 9. 回溯能力边界

本版只提供最低限度回溯：

1. 能知道当前 active 由哪次 `publish_version` 激活
2. 能知道这次发布对应的 `source_batch`
3. 能知道某个对象来自哪个 `origin_batch_id`
4. 能知道某次 `mining_run` 处理了哪些文档
5. 能通过 `source_refs_json` / `source_offsets_json` 回到原文

本版不提供：

1. 任意历史版本在线查询
2. 任意历史版本完整对象集合重建
3. segment 级精细版本演化树

## 10. 回退能力边界与后续升级路径

### 10.1 为什么当前版本不支持“任意历史版本在线回退”

本版有意不支持：

```text
把系统从当前版本直接切换回任意历史版本
```

原因不是忽略这个需求，而是当前设计明确放弃了以下高复杂度机制：

1. 每个版本一套物理全量快照
2. 每条对象带 `created_in_version / retired_in_version`
3. `document_revision` / Git 式增量版本模型

因此，本版的 Asset DB 只保存：

- 当前 active 资产
- 发布事件元信息
- 数据来源追溯信息

而不保存：

- 每个历史版本对应的完整对象集合

所以当前系统能回答：

- 当前 active 是哪次发布
- 这次发布来自哪个 batch / run
- 某条数据来自哪个文档、哪个 batch、哪次 run

但不能直接回答：

- “v3 当时完整包含哪些 retrieval_units / segments / relations”
- “把当前系统一键切回 v3 在线服务”

### 10.2 当前版本实际支持的“回退”

本版支持的是**最低限度工程回退**，不是在线多版本回退。

具体包括：

#### A. 发布事件回退

能记录：

- 当前 active publish_version
- 上一个 active publish_version
- 每次发布的构建摘要
- 每次发布的来源 batch / run

#### B. 数据来源回退

能回溯：

- retrieval unit -> raw segment
- raw segment -> raw document
- raw document -> source batch
- source batch -> mining run

#### C. 工程包级回退

如果每次成功发布时额外归档一份发布包，例如：

```text
asset_bundle_v12.sqlite
```

那么系统虽然不支持在线版本切换，但可以支持：

- 运维层面的上一版恢复
- 故障时人工回切
- 离线比对某次历史发布

建议在 `asset_publish_versions.metadata_json` 中预留：

```json
{
  "bundle_path": "storage://asset-bundles/v12.sqlite",
  "bundle_checksum": "sha256:..."
}
```

这属于当前版本最推荐的最小回退补充方案。

### 10.3 当前不做在线历史版本回退的收益

放弃完整在线回退，换来的收益是：

1. Asset 主表不需要挂版本字段
2. Serving 查询保持简单
3. 不需要为每个版本重复存储对象或挂载关系
4. 不需要引入 revision / delta 图模型
5. Mining / Serving / LLM 三方可以先把主链路跑稳

这符合 1.1 当前阶段的首要目标。

### 10.4 后续升级路径

如果未来确实需要更强版本能力，建议按以下顺序升级，而不是一次性引入最复杂方案。

#### 升级路径 1：发布包归档（推荐最先做）

目标：

- 增加可运维回退能力
- 不改变当前查询模型

做法：

1. 每次新版本发布成功后，导出一份只读资产包
2. 在 `asset_publish_versions.metadata_json` 中记录包路径
3. 回退时通过替换 active 资产包恢复

优点：

- 影响最小
- 查询模型不变
- 最符合当前需求

#### 升级路径 2：active + previous 双快照

目标：

- 支持快速回退到上一个版本
- 不支持任意历史版本

做法：

1. 只保留当前 active 和上一版 previous 两份只读资产包
2. `publish_versions` 中记录两者关系
3. 运维回切只支持 `active -> previous`

优点：

- 比任意版本系统简单很多
- 对绝大多数线上回退场景已足够

#### 升级路径 3：版本挂载模型

目标：

- 支持在线多版本视图
- 支持有限历史版本读取

做法：

引入：

```text
asset_publish_version_documents
asset_publish_version_segments
asset_publish_version_retrieval_units
```

注意：

- 这是后续版本才考虑的能力
- 只有在明确需要在线读历史版本时才有必要

#### 升级路径 4：revision / Git 式模型

目标：

- 支持更细粒度历史演化管理
- 支持对象复用 + 版本增量表达

做法：

引入：

```text
document_revision
delta / parent chain
```

注意：

- 这是复杂度最高的一档
- 当前 1.1 不应提前引入

### 10.5 当前推荐决策

当前推荐保持：

```text
在线系统不支持任意历史版本回退
```

并建议补充：

```text
每次成功发布时归档一份只读资产包
```

这样就能在不复杂化主数据库模型的前提下，获得：

1. 当前态稳定服务
2. 最小发布边界
3. 最小工程回退能力
4. 后续明确升级入口

## 12. 对三位员工的职责划分

### `claude-mining`

负责：

- Mining Runtime DB
- staging 资产构建
- 发布切换
- `asset_raw_*`
- `asset_retrieval_units`

不负责：

- Serving 查询逻辑
- LLM Runtime 表

### `claude-serving`

负责：

- 从 active 资产读取
- FTS / BM25 / relation expansion
- ContextPack 输出

不负责：

- 写 Asset DB
- 写 Mining Runtime DB
- 写 LLM Runtime DB

### `claude-llm`

负责：

- `agent_llm_*` 全套表
- 队列 / provider / parser / JSON schema 校验

不负责：

- 写 Asset 资产表
- 写 Mining Runtime 表

## 13. 最终判断

这版是当前最合适的 1.1 数据库基线：

1. 没有引入 `document_revision`
2. 没有引入每表 start/end version
3. 没有做物理全量快照
4. 保留了最小发布边界
5. 让 Serving 查询保持简单
6. 让 Mining 能做断点续跑和过程诊断
7. 为后续继续演进预留空间

这就是当前推荐的最终版。
