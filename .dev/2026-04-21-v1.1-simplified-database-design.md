## 1. 设计目标

| 需求 | 说明 |
|---|---|
| 增量挖掘 | 新数据来了只处理新增/变更文档，跳过未变文档 |
| 断点续跑 | 挖掘过程中断后可从断点恢复，不重做已完成的文档 |
| 旧数据清理 | 批次失效后，清理只属于失效批次的文档及其下游资产 |
| Serving 查询简单 | 直接查所有 retrieval_units，不需要版本过滤 |
| 不重复存储 | 未变更的文档数据不重复落盘 |

## 2. 与 Codex 方案的核心差异

| 点 | Codex 三层方案 | 本方案 |
|---|---|---|
| 版本模型 | publish_version + document_revision + 对象层 | 无版本模型，Asset DB 只存当前状态 |
| 每表挂版本 | asset 表带 publish_version_id | Asset 表不带版本字段 |
| 回溯能力 | 支持 | 不支持（v1.1 不需要） |
| 表数量 | 14 张（含 mining runtime） | 10 张（Asset 7 + Mining Runtime 3） |
| 挖掘态 | mining_runs / mining_run_documents / mining_stage_events | mining_runs / mining_run_documents（简化） |
| Serving 查询 | 需过滤 active publish_version | 直接查，无版本过滤 |

## 4. 增量挖掘流程

```text
用户触发: mining run --input /path/to/new/data

  ┌─ Phase 0: 初始化 ──────────────────────────────┐
  │  创建 mining_run 记录 (status: running)          │
  │  创建 source_batch 记录                          │
  └────────────────────────────────────────────────┘

  ┌─ Phase 1: 发现与分类 ───────────────────────────┐
  │  扫描输入目录，收集文件列表                        │
  │  for each file:                                  │
  │    doc_key = relative_path                       │
  │    hash = sha256(file_content)                   │
  │                                                  │
  │    # 查 Asset DB 中是否已有该 document_key        │
  │    existing = asset_raw_documents                │
  │                  .find(document_key = doc_key)    │
  │                                                  │
  │    if not existing:                              │
  │      → action: NEW                              │
  │    elif existing.content_hash == hash:            │
  │      → action: SKIP (内容未变)                    │
  │      → 仍然在 asset_batch_documents 挂载当前批次   │
  │    else:                                         │
  │      → action: UPDATE (内容变更)                  │
  │                                                  │
  │    写入 mining_run_document (status: pending)     │
  └────────────────────────────────────────────────┘

  ┌─ Phase 2: 增量处理 ─────────────────────────────┐
  │  for each run_doc where action in (NEW, UPDATE): │
  │                                                  │
  │    if action == UPDATE:                          │
  │      # 先删除旧数据 (在事务内)                     │
  │      delete asset data for existing document      │
  │                                                  │
  │    update run_doc status → processing             │
  │                                                  │
  │    解析 → 切片 → 构建关系 → 生成 retrieval_units  │
  │    每个阶段完成时记录 stage_event                  │
  │                                                  │
  │    if 全部阶段成功:                                │
  │      在 Asset DB 事务中写入:                      │
  │        - raw_document                            │
  │        - raw_segments                            │
  │        - raw_segment_relations                   │
  │        - retrieval_units                         │
  │        - batch_documents 挂载                     │
  │      update run_doc status → committed            │
  │    else:                                         │
  │      update run_doc status → failed               │
  │      记录 error 信息                              │
  └────────────────────────────────────────────────┘

  ┌─ Phase 3: 清理旧批次 ───────────────────────────┐
  │  if 用户指定清理旧批次:                            │
  │    mark old batches as inactive                   │
  │                                                  │
  │    # 找出只属于 inactive 批次的文档                 │
  │    orphan_docs = asset_batch_documents             │
  │      .group_by(document_id)                       │
  │      .having(all batches are inactive)             │
  │                                                  │
  │    for each orphan_doc:                           │
  │      cascade delete:                              │
  │        retrieval_units → relations                │
  │        → segments → document → batch_documents    │
  └────────────────────────────────────────────────┘

  ┌─ Phase 4: 收尾 ─────────────────────────────────┐
  │  重建 FTS 索引                                    │
  │  更新 mining_run status → completed               │
  │  统计摘要写入 mining_run.metadata_json             │
  └────────────────────────────────────────────────┘
```

## 5. 断点续跑

### 5.1 状态机

每个文档在 mining_run_documents 中的状态流转：

```text
pending ──→ processing ──→ committed
                 │              ↑
                 │     (重试/续跑时跳过)
                 ▼
              failed ──→ (重试时可重新设为 pending)
```

### 5.2 续跑逻辑

```text
用户触发: mining resume --run <run_id>

  读取 mining_run 记录，确认 status = interrupted/failed

  读取该 run 下所有 mining_run_documents:
    status = committed  → 跳过 (已写入 Asset DB)
    status = pending    → 正常处理
    status = processing → 重新处理 (上次中断时可能已部分写入)
    status = failed     → 根据策略决定重试或跳过

  处理流程与正常挖掘相同，只是跳过已 committed 的文档
```

### 5.3 为什么需要 mining_run_stage_events

虽然一个文档"未 committed 就从头重做"在逻辑上是可行的，但记录每个阶段的完成情况有以下好处：

1. **诊断**：如果某个文档反复失败，可以看到是在哪个阶段失败的
2. **耗时统计**：记录每个阶段的耗时，方便优化瓶颈
3. **未来优化**：v2 可以做到真正的阶段级续跑（比如只重新生成 retrieval_units 而不重新 parse）

v1.1 的 stage_event 是轻量的追加写入，不影响主流程性能。

## 7. 关键操作

### 7.1 增量挖掘（正常流程）

```text
# Phase 1: 发现
扫描输入目录 → 计算 content_hash → 对比 Asset DB 中的文档

# Phase 2: 处理
对每个待处理文档:
  1. 在 Mining Runtime DB 中标记 processing
  2. 在内存中完成 parse → segment → enrich → relations → retrieval_units
  3. 每个阶段记录 stage_event
  4. 如果是 UPDATE: 在 Asset DB 事务中先删旧数据再写新数据
  5. 如果是 NEW: 直接写入 Asset DB
  6. 写入 asset_batch_documents 挂载
  7. 标记 committed

# Phase 3: 清理（可选）
标记旧批次 inactive → 找孤立文档 → 级联删除
```

### 7.2 断点续跑

```text
# 读取上次中断的 run
run = mining_runs.find(run_id)

# 找出未完成的文档
pending_docs = mining_run_documents
    .where(run_id = run_id, status IN ('pending', 'processing', 'failed'))

# 对 pending / processing 文档：重新处理
# 对 failed 文档：根据策略决定是否重试

# 继续正常的 Phase 2 和 Phase 3
```

### 7.3 Serving 查询

```text
# 无版本过滤，直接查
SELECT * FROM asset_retrieval_units
WHERE search_text MATCH ?
ORDER BY weight DESC
LIMIT 20

# 命中后扩展上下文
SELECT target.* FROM asset_raw_segment_relations rel
JOIN asset_raw_segments target ON target.id = rel.target_segment_id
WHERE rel.source_segment_id IN (命中 segments)
AND rel.relation_type IN ('previous', 'next', 'same_section', 'section_header_of')
```

### 7.4 清理旧数据

```text
# 1. 标记批次为 inactive
UPDATE asset_source_batches SET is_active = 0 WHERE id = ?

# 2. 找出只属于 inactive 批次的文档
SELECT d.id
FROM asset_raw_documents d
JOIN asset_batch_documents bd ON bd.document_id = d.id
JOIN asset_source_batches b ON b.id = bd.batch_id
GROUP BY d.id
HAVING COUNT(*) = SUM(CASE WHEN b.is_active = 0 THEN 1 ELSE 0 END)

# 3. 级联删除
DELETE FROM asset_retrieval_units WHERE raw_document_id IN (orphan_ids)
DELETE FROM asset_raw_segment_relations WHERE source_segment_id IN (segments of orphan docs)
DELETE FROM asset_raw_segments WHERE raw_document_id IN (orphan_ids)
DELETE FROM asset_batch_documents WHERE document_id IN (orphan_ids)
DELETE FROM asset_raw_documents WHERE id IN (orphan_ids)
```

## 10. 数据库文件布局（dev 模式 SQLite）

```text
data/
  <corpus_name>/
    mining_runtime.sqlite    ← Mining Runtime DB
    asset.sqlite             ← Knowledge Asset DB
    fts_index/               ← FTS 相关（可选独立目录）
```

Mining Runtime 和 Asset 分文件的原因：
1. Mining Runtime 是临时态，可以随时重建
2. Asset 是持久态，需要备份和保护
3. Mining 写入时不影响 Serving 读取性能

## 11. 设计原则总结

1. **Asset DB 无版本** — 只存当前有效状态，不做回溯
2. **content_hash 驱动增量** — hash 不变就跳过
3. **文档粒度原子替换** — UPDATE 时整个文档的数据在事务中替换
4. **批次-文档多对多** — 安全清理，不误删共享文档
5. **Mining Runtime 独立** — 断点续跑、诊断、统计
6. **Serving 无感** — 不需要知道版本，直接查当前数据
7. **canonical 降级** — 保留代码和表，但不走主路径
