SECTION 03

两个卡点

从"能做 2 个场景"到"能做 N 个场景"，中间有两个核心阻塞。

01

业务图谱怎么构建

  • 场景发现难：用户意图是开集，不知道哪些场景高频、值得优先投入
  • Schema 不稳：业务图谱的数据结构尚未定型，不同场景差异大
  • 知识来源散：知识分散在 MOP/工单、产品文档、现网脚本、专家经验四类来源中
  • 生成/核查割裂：业务图谱是唯一能把两条链路统一到底座的层
AND
02

Skill 怎么规模化

  • 手工拼装：当前每个场景都独立写 Skill，APN 一个、DPI 一个，无法复用
  • 知识内嵌 vs 外挂：什么应放在 Skill 内部、什么应沉淀为后端服务，边界不清
  • 缺生成机制：没有"知识召回 → 流程编排 → 工具注入"的能力生成流水线
  • 场景包概念缺失：还没有"通用 Runtime + 场景包"的分层架构

SECTION 04

业务图谱构建路线

核心思路：不凭空设计，而是从真实作业记录中挖掘场景、归纳结构、填充知识。

2

场景建模 — 定义业务图谱 Schema

对 Top-N 场景，建立稳定的结构化表达。每个场景的业务图谱至少包含以下字段。

核查义务

3

知识填充 — 四源汇聚

业务图谱的知识不是单源产物，需要从四类来源汇聚填充。LLM 辅助抽取，专家确认。

来源 1 · MOP / 工单

提供特性适用条件、组合约束、参数规范
来源 3 · 现网脚本 / 样板局点
提供真实实现路径、常用范式、关键变参
来源 4 · 核查规则 / 专家经验
提供"什么算对/什么算错"的业务语义

三个关键技术机制

知识召回

Skill 不内嵌图谱内容，而是持有"如何访问图谱"的逻辑。 运行时通过知识服务 / 图谱服务 API，按场景上下文检索所需知识。

Skill → 图谱服务(query, context) → 业务图谱片段

流程编排

Skill 内部的多轮交互流程用声明式描述（状态机 / DAG），而非硬编码。 每个场景包定义自己的流程模板，Runtime 负责执行。

场景包.yaml → flow_dag → Runtime.execute()

工具注入

后端能力沉淀为 CLI / Service，注册到 Runtime 工具表。 场景包声明"我需要哪些工具"，Runtime 负责注入。

场景包.tools = [instantiate, generate, check]

AGENT 层

SKILL 层

后端服务
图谱实例化 · 模板生成 · 核查 · 语义比对 · CLI / Service 注册
三层图谱

RAG · 向量检索 · ContextPack · 支撑图谱构建的知识来源
核心价值：图谱解决"知道什么"，Skill 解决"怎么跑起来"。当业务图谱填补了三层图谱的缺失层，Skill Runtime 建立了规模化扩展机制，配置生成将从"手工作坊"演进为"可扩展的工程体系"。配置生成与配置核查将共用同一套图谱底座，而非继续割裂推进。
