KEY CONCEPTS

两个关键概念：业务图谱 & Skill

它们是什么、有什么区别、缺了哪个都不行。

业务图谱 = 场景知识资产

它回答：这个场景有什么选项、该怎么选、什么组合算对、什么算错。

以 APN 开通为例，上面那棵配置树就是业务图谱：
· 地址分配有 6 种方式（OR节点，选一个）
· 鉴权有 4 种模式，不同模式对应不同配置（AND节点）
· 工厂场景选了 UDM + 不透明 + IPSec → 组合约束全部满足
知识来源：MOP工单 + 产品文档 + 现网脚本 + 专家经验

Skill = 执行流程模板

它回答：Agent 在这个场景里怎么引导用户、什么时候调用后端服务、怎么组织输出。

以 APN 场景为例，左侧那个对话流程就是 Skill：
· 先确认场景类型，再逐步问接入方式、地址分配、鉴权、地址类型
· 每步根据业务图谱推荐选项，用户做选择题而非填空题
· 所有槽位确认后，调用后端服务生成配置并核查
关键：Skill 运行时动态读取业务图谱，不把知识写死在流程里
一句话关系：业务图谱定义"有什么、怎么选"，Skill 定义"按什么流程引导用户去选"。Skill 运行时通过服务层读取业务图谱。二者缺一不可——没有图谱，Skill 不知道问什么；没有 Skill，图谱只是静态知识摆在那。

业务图谱和 Skill 怎么从当前状态走到目标态。核心定位：图谱是资产，服务是能力，Skill是流程。

卡点二：Skill 构建

当前：手工拼装
APN 一个 Skill，DPI 一个 Skill，知识全内嵌
目标：Runtime + 场景包
通用 Runtime 负责意图识别、槽位管理、多轮澄清、服务编排
场景包只装槽位定义 + 流程模板 + 工具映射
Skill 通过服务层访问图谱 → 保证数据安全
三个关键机制：
知识召回 — 运行时按场景检索图谱，不内嵌
流程编排 — 声明式状态机，场景包定义模板
工具注入 — 后端能力注册到 Runtime，按需注入
核心价值：图谱解决"知道什么"，Skill 解决"怎么跑起来"，服务层保证"安全可控"。业务图谱填补三层图谱缺失层后，配置生成从"手工作坊"演进为"可扩展工程体系"。生成与核查共用图谱底座，不再割裂。
