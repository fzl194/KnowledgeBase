# 面向 Agent 的云核心网 Graph-RAG 后端服务设计说明

## 1. 文档目的

本文档用于承载当前项目后续演进的完整上下文。目标是让用户可以拿着这份文档，与任何 Agent、研发人员或架构讨论对象继续沟通，而不需要重复解释项目背景、业务意图和关键设计取舍。

本文档不关注具体代码实现，也不要求立即落到某个技术框架。它关注的是：在现有“电信语义知识操作系统”项目基础上，如何从“本体抽取系统”进一步补齐“真实业务使用层”，尤其是如何让 Agent 能够通过 Skill 调用后端服务，从云核心网知识库中灵活检索、理解和回答工程问题。

## 2. 业务背景

当前项目已经具备较强的知识加工能力，核心包括：

- 原始文档接入、清洗、去重和质量门控。
- 文档切分为知识片段。
- 段落语义类型识别。
- 段落间 RST 关系识别。
- 本体对齐。
- 候选本体演化。
- 事实三元组抽取。
- 事实去重和冲突检测。
- Neo4j 图投影。
- 向量检索和若干语义算子。

这些能力更偏向“知识抽取”和“知识治理”。它们能把文档逐步加工成本体节点、事实、证据和图数据库结构。但从真实业务使用视角看，仅有这些能力还不够。

真实业务场景中，专家或工程师通常不会问：

```text
请查询某个本体节点的三元组。
```

他们更可能问：

```text
ADD APN 命令怎么写？
新增一个 APN/DNN 需要配置哪些地方？
PDU Session 建立失败该怎么排查？
SMF 和 UPF 的 N4 配置有什么前置条件？
修改某个 APN 会不会影响现网业务？
某个参数是什么意思，什么时候要配？
某个告警对应哪些可能原因？
```

因此，项目需要从“知识抽取后台”演进为“面向 Agent 的云核心网知识后端”。Graph-RAG 在这里不是一个给人直接操作的搜索界面，而是 Agent 背后的知识服务能力。

## 3. 用户的真实目标

用户的最终目标可以概括为：

```text
构建一个云核心网知识后端服务。
该服务被 Skill 封装。
Skill 被 Agent 调用。
Agent 借助这个 Skill 理解和使用云核心网知识库。
```

系统顶层依赖已有 Agent 框架。用户不打算重新实现 Agent 框架，而是重点构建中间的知识服务能力。

整体关系如下：

```text
Agent 层
  依赖已有 Agent 框架
  负责理解用户意图、选择工具、组织最终回答

Skill 层
  依赖云核心网知识后端服务
  负责把 Agent 的业务意图翻译成稳定工具调用
  负责定义调用策略、答案格式和追问规则

后端服务层
  当前重点建设对象
  负责文档加工、索引、Graph-RAG 检索、上下文组装、证据返回
```

用户希望最终形成类似 Skill 的能力包，让 Agent 可以通过它使用云核心网知识库。例如业务专家问“ADD APN 命令怎么写”，Agent 能够灵活调用后端服务，查到命令模板、参数含义、配置示例、前置条件、注意事项、版本差异和来源证据，然后组织成工程师能理解和执行的回答。

## 4. 当前项目的定位与局限

当前项目更像一个“语义知识加工系统”或“本体治理系统”。它的优势是：

- 对文档进行结构化加工。
- 构建五层本体模型。
- 抽取事实、证据和关系。
- 支持知识来源追溯。
- 支持本体演化和候选词治理。
- 支持图数据库和向量检索。

但它在“业务使用层”还有明显不足：

- Agent 不应该直接理解 pipeline、stage、表结构、Cypher 或 SQL。
- 现有检索接口偏底层，例如通用语义搜索、本体查找、路径推理等。
- 缺少面向业务意图的接口，例如“查命令写法”“查配置流程”“查故障处理”“查影响范围”。
- 缺少面向 Agent 的上下文包结构。
- 缺少针对云核心网场景的命令、参数、配置对象、网元、流程和版本差异建模。
- Graph-RAG 的使用方式还没有产品化：现有系统能建图，但尚未明确 Agent 在实际问答中如何使用这张图。

因此，后续重点不是继续盲目增强本体抽取，而是补齐“Agent 使用层所需的后端服务能力”。

## 5. 顶层架构建议

建议把系统分为三层。

### 5.1 Agent 层

Agent 层使用已有 Agent 框架，不是本项目重点建设内容。

Agent 的职责包括：

- 理解用户自然语言问题。
- 判断用户意图。
- 选择合适 Skill 或工具。
- 根据工具返回的上下文包生成最终回答。
- 在信息不足时向用户追问。
- 在存在多个版本、厂家或场景时明确不确定性。

Agent 不应该直接访问数据库，也不应该直接处理复杂 Graph-RAG 策略。

### 5.2 Skill 层

Skill 层是 Agent 和后端之间的桥。

Skill 不承载完整知识库，也不负责重型检索。它应该承载：

- 后端服务地址和认证方式。
- 可调用工具的定义。
- 每个工具适合什么业务意图。
- 何时调用哪个工具。
- 工具返回结果如何解释。
- 答案应该按什么业务格式输出。
- 信息缺失时如何追问。

Skill 的职责是“教 Agent 如何使用云核心网知识后端”，而不是把后端逻辑塞进 Skill。

### 5.3 后端服务层

后端服务层是当前最需要重点建设的部分。

它负责：

- 文档导入和加工。
- 文档清洗、切分、去重。
- 段落图构建。
- 向量索引。
- 命令、参数、流程、故障、约束等业务对象抽取或轻量标注。
- Graph-RAG 检索。
- 图扩展。
- 重排序。
- 上下文组装。
- 证据返回。
- 版本、厂家、产品、场景等过滤。
- 面向 Agent 的稳定 API。

一句话概括：

```text
后台做重活，Skill 做编排说明，Agent 做自然语言交互和最终回答。
```

## 6. Graph-RAG 在本系统中的真实作用

Graph-RAG 不应该被理解成“把所有知识都抽成三元组之后再问答”。在云核心网业务中，很多有价值的知识并不天然适合抽成单一三元组。

例如命令手册中的内容通常包括：

- 命令名。
- 命令格式。
- 参数表。
- 参数取值范围。
- 默认值。
- 配置示例。
- 注意事项。
- 前置条件。
- 适用版本。
- 涉及网元。
- 对业务的影响。

这些内容更适合通过文档结构、段落关系、命令对象、参数对象和证据片段组合使用。

因此，建议把 Graph-RAG 分成几类图来理解。

### 6.1 文档图

表达文档结构：

```text
SourceDocument
  -> Section
  -> Segment
```

用途：

- 保留原始文档层级。
- 让 Agent 能知道某段知识来自哪个章节。
- 支持按章节扩展上下文。

### 6.2 段落关系图

表达段落之间的上下文关系：

```text
Segment -> Segment
```

关系可以来自 RST 或结构邻接，例如：

- Sequence。
- Explanation。
- Elaboration。
- Condition。
- Problem-Solution。
- Evidence。
- Background。

用途：

- 用户命中一个命令段时，自动扩展到参数段、示例段、注意事项段。
- 用户命中一个故障段时，自动扩展到原因段、处理步骤段。
- 避免单段召回导致上下文断裂。

### 6.3 命令图

表达业务操作知识：

```text
Command
  -> Parameter
  -> ValueConstraint
  -> Example
  -> Warning
  -> RelatedNetworkFunction
```

用途：

- 回答“命令怎么写”。
- 回答“参数是什么意思”。
- 回答“这个参数是否必填”。
- 回答“有没有示例”。
- 回答“配置前要注意什么”。

这是云核心网业务最有价值的图之一，应优先建设。

### 6.4 网元与流程图

表达云核心网对象和流程关系：

```text
AMF / SMF / UPF / UDM / PCF / NRF / AUSF
N1 / N2 / N3 / N4 / N6 / N7 / N10 / N11
PDU Session / DNN / APN / Slice / QoS Flow / PFCP Session
```

用途：

- 回答流程类问题。
- 回答某个配置涉及哪些网元。
- 回答某个故障可能发生在哪个接口或流程。
- 支持影响分析。

### 6.5 证据图

表达答案依据：

```text
AnswerClaim
  -> EvidenceSegment
  -> SourceDocument
```

用途：

- 给 Agent 提供可引用来源。
- 让回答不只是“模型知道”，而是“基于哪些文档片段得出”。
- 支持业务专家核查。

## 7. 后端服务应该提供的能力

后端不应只提供一个通用搜索接口。真实 Agent 使用时，需要意图化 API。

### 7.1 通用检索服务

用于兜底查询。

能力包括：

- 关键词检索。
- 向量检索。
- 标题增强检索。
- 语义重排序。
- 来源过滤。
- 版本过滤。
- 厂家过滤。
- 场景过滤。
- 图扩展。

适合问题：

```text
查询某个术语。
解释某个概念。
找相关文档片段。
```

### 7.2 命令检索服务

这是第一优先级。

输入可以是：

```text
ADD APN
MOD APN
ADD DNN
SMF 选择策略配置命令
UPF N4 配置命令
```

输出应该尽量结构化：

- 命令名。
- 命令模板。
- 参数列表。
- 参数含义。
- 是否必填。
- 取值范围。
- 默认值。
- 示例。
- 前置条件。
- 注意事项。
- 适用厂家、产品、版本。
- 来源证据。
- 不确定性和需要追问的信息。

### 7.3 流程检索服务

用于回答“怎么做”。

例如：

```text
如何新增一个 APN？
如何新增一个 DNN？
如何配置 SMF 到 UPF 的 N4？
如何开通一个 5G 专网切片？
如何排查 PDU Session 建立失败？
```

输出应该包括：

- 操作目标。
- 适用场景。
- 涉及网元。
- 前置条件。
- 配置步骤。
- 验证步骤。
- 回退或风险。
- 相关命令。
- 证据来源。

### 7.4 约束与前置条件检索服务

用于回答：

```text
执行这个配置前需要满足什么？
这个参数什么时候能配？
这个命令有什么限制？
这个配置和版本有没有关系？
```

云核心网配置高度依赖版本、厂家、License、组网模式、漫游场景、切片场景等，因此约束检索非常重要。

### 7.5 故障与排障检索服务

用于回答：

```text
PDU Session 建立失败怎么查？
SMF 选择失败有哪些原因？
UPF N4 不通怎么办？
NRF 注册失败怎么处理？
用户无法访问某个 APN/DNN 怎么排查？
```

输出应该包括：

- 故障现象。
- 可能原因。
- 排查路径。
- 涉及网元。
- 涉及接口。
- 相关命令。
- 验证方法。
- 证据来源。

### 7.6 影响分析服务

用于回答：

```text
修改 APN 会影响哪些业务？
修改 DNN 配置会影响哪些用户？
修改 UPF 选择策略有什么风险？
修改 SMF 配置会影响哪些流程？
```

影响分析是高价值能力，但可以在命令检索和流程检索之后建设。

## 8. 后端内部推荐流程

一次 Agent 检索请求，后端内部可以按以下流程处理。

### 8.1 意图识别

判断用户问题属于哪类：

- command_usage：命令写法。
- procedure：配置流程。
- troubleshooting：故障排查。
- constraint：前置条件或限制。
- impact：影响分析。
- concept_explain：概念解释。
- compare：对比。
- evidence_lookup：查依据。

### 8.2 查询归一化

云核心网中同一对象可能有多种说法：

```text
APN / DNN
P-GW / SMF / UPF
APN 配置 / DNN 配置 / 签约数据
ADD APN / 新增 APN / 创建 APN
PDU Session / 会话建立
N4 / PFCP
```

后端需要维护领域同义词、命令别名和版本映射。

### 8.3 候选召回

召回不应只依赖向量。

建议组合：

- 命令名精确匹配。
- 标题匹配。
- 关键词匹配。
- 向量召回。
- 文档章节召回。
- 图邻居召回。
- 同义词扩展召回。

例如用户问 `ADD APN`，命令名精确匹配和标题匹配应该优先于普通语义向量。

### 8.4 图扩展

命中一个段落后，不应只返回这个段落。

应该根据图关系扩展：

- 前一段和后一段。
- 同章节下的参数段。
- 示例段。
- 注意事项段。
- 条件段。
- 故障处理段。
- 与该命令关联的网元和流程。

### 8.5 重排序

根据业务意图重新排序。

例如命令查询时，优先级可以是：

```text
命令名精确命中
> 标题包含命令
> 参数表
> 配置示例
> 注意事项
> 普通解释段
```

同时考虑：

- 来源权威性。
- 版本匹配度。
- 厂家匹配度。
- 文档时间。
- 段落完整性。
- 是否有示例。
- 是否能回答用户问题。

### 8.6 上下文组装

最终不要只返回裸段落列表，而要返回 Agent 可直接使用的上下文包。

上下文包应该包括：

- 用户原始问题。
- 检测到的意图。
- 查询归一化结果。
- 命中的关键对象。
- 结构化答案素材。
- 证据片段。
- 来源文档。
- 不确定性。
- 建议追问。
- 可选的回答草稿。

## 9. Skill 应该如何设计

Skill 的核心不是存知识，而是定义 Agent 如何调用后端。

Skill 中应描述：

### 9.1 可用工具

例如：

```text
search_cloud_core_knowledge
get_command_usage
get_procedure
get_troubleshooting
get_constraints
get_impact
assemble_cloud_core_context
```

### 9.2 工具选择策略

例如：

- 用户问“命令怎么写”“参数怎么填”“有没有示例”，优先调用 `get_command_usage`。
- 用户问“怎么开通”“怎么新增”“怎么配置”，优先调用 `get_procedure`。
- 用户问“失败”“不通”“报错”“告警”，优先调用 `get_troubleshooting`。
- 用户问“是否影响业务”“风险是什么”，优先调用 `get_impact`。
- 用户问“依据是什么”“出处在哪里”，调用证据或上下文组装工具。

### 9.3 答案格式

对于命令类问题，Agent 应按业务格式回答：

```text
适用场景
命令模板
参数说明
配置示例
前置条件
注意事项
常见错误
来源依据
需要确认的信息
```

对于流程类问题：

```text
目标
适用范围
涉及网元
前置条件
操作步骤
验证方法
回退建议
风险提示
来源依据
```

对于故障类问题：

```text
现象确认
可能原因
排查路径
涉及网元/接口
常用命令
处理建议
来源依据
```

### 9.4 追问规则

云核心网问题经常缺少关键上下文。Skill 应告诉 Agent 何时追问。

常见缺失项：

- 厂家。
- 产品。
- 版本。
- EPC 还是 5GC。
- APN 还是 DNN。
- 涉及哪个网元。
- SA 还是 NSA。
- 是否涉及切片。
- 是否涉及漫游。
- 是否是现网变更还是新开局。

例如用户只问“ADD APN 命令怎么写”，Agent 可以先回答通用结果，同时提示：

```text
这个命令在不同厂家、网元和版本上可能不同。当前问题还缺少厂家和产品版本。如果你能提供这些信息，我可以进一步收敛到精确命令。
```

## 10. Agent 的典型使用链路

以用户问题为例：

```text
ADD APN 命令怎么写？
```

推荐链路：

```text
Agent 判断意图：命令写法
  ↓
Skill 选择工具：get_command_usage
  ↓
后端执行：
  - 归一化 ADD APN
  - 查命令名精确匹配
  - 查 APN/DNN 相关段落
  - 向量召回配置示例
  - 沿 RST 扩展参数表和注意事项
  - 按厂家、版本、来源重排序
  - 返回上下文包
  ↓
Agent 检查是否缺少厂家/版本
  ↓
Agent 组织回答：
  - 命令模板
  - 参数说明
  - 示例
  - 前置条件
  - 注意事项
  - 需要确认的信息
  - 来源依据
```

这条链路体现了一个核心原则：

```text
Agent 负责表达和决策，后端负责检索和证据，Skill 负责连接两者。
```

## 11. 最小可用版本建议

不建议一开始就追求完整本体抽取闭环。为了快速让 Agent 可用，建议先做最小 Graph-RAG。

最小版本包括：

### 11.1 数据加工

保留并重点使用：

- Stage 1：文档清洗。
- Stage 2：文档切分、段落类型、RST 关系。

### 11.2 索引

建立：

- segment 向量索引。
- 标题索引。
- 命令名关键词索引。
- RST 段落关系索引。
- 来源元数据索引。

### 11.3 检索

优先实现：

- 通用混合检索。
- 命令检索。
- 上下文组装。

### 11.4 Skill

先提供少量工具：

- `get_command_usage`
- `search_cloud_core_knowledge`
- `assemble_cloud_core_context`

### 11.5 能力边界

第一版不强求：

- 完整本体演化。
- 自动事实抽取全覆盖。
- 所有配置流程自动结构化。
- 全量影响分析。

第一版的目标应该是：

```text
Agent 能基于云核心网资料，可靠回答命令写法、参数含义、配置示例、注意事项，并带证据。
```

## 12. 完整版本演进方向

当最小版本可用后，再逐步增强。

### 12.1 命令结构化

从文档中进一步抽取：

- Command。
- Parameter。
- ParameterGroup。
- Example。
- Constraint。
- Warning。
- VersionApplicability。

### 12.2 流程结构化

抽取：

- Procedure。
- Step。
- Precondition。
- Validation。
- Rollback。
- RelatedCommand。
- RelatedNF。

### 12.3 故障知识结构化

抽取：

- Symptom。
- Cause。
- Check。
- Fix。
- Verification。
- RelatedAlarm。

### 12.4 本体与业务图融合

将已有五层本体用于增强推理，而不是替代业务检索。

例如：

- APN/DNN 归入数据网络接入对象。
- SMF/UPF/PCF/UDM 归入云核心网网元。
- PDU Session Establishment 归入机制/流程。
- N4/PFCP 归入接口/协议。
- APN/DNN 配置命令关联到对应网元和流程。

### 12.5 影响分析

基于业务图回答：

- 某个命令影响哪些配置对象。
- 某个配置对象影响哪些流程。
- 某个流程涉及哪些网元。
- 某个网元异常影响哪些业务。

## 13. 关键设计原则

### 13.1 不要让 Agent 直接查底层表

Agent 不应该知道：

- `documents`
- `segments`
- `segment_tags`
- `facts`
- `evidence`
- `t_rst_relation`
- Neo4j Cypher
- pgvector SQL

这些属于后端实现细节。

### 13.2 不要把重逻辑塞进 Skill

Skill 应保持轻量。它应该定义工具、策略和答案规范，而不是实现复杂检索、重排序和图遍历。

### 13.3 后端 API 要业务语义化

比起：

```text
semantic_search(query)
```

更适合 Agent 的是：

```text
get_command_usage(command, filters)
get_procedure(task, filters)
get_troubleshooting(symptom, filters)
get_constraints(object, action, filters)
```

### 13.4 证据必须一等公民

业务专家不会只接受“模型说”。回答应能追溯：

- 哪个文档。
- 哪个章节。
- 哪个片段。
- 哪个版本。
- 哪个厂家。

### 13.5 版本和场景必须显式化

云核心网知识高度依赖上下文。后端和 Skill 都应显式处理：

- 厂家。
- 产品。
- 版本。
- 组网场景。
- 业务场景。
- 现网变更还是新开局。

### 13.6 Graph-RAG 的价值在于上下文扩展

Graph-RAG 不是为了展示图，而是为了帮助 Agent 找到“一个问题所需的一组相关证据”。

例如命中命令段后，图应该帮助找到：

- 参数段。
- 示例段。
- 注意事项段。
- 前置条件段。
- 故障段。
- 相关网元段。

## 15. 对当前项目的具体建议

当前项目不需要推翻，应在现有基础上新增“Agent Knowledge Backend”层。

建议新增一个逻辑模块，例如：

```text
agent_knowledge
```

该模块负责：

- 面向 Agent 的 API。
- 意图识别。
- 云核心网查询归一化。
- 混合检索。
- 图扩展。
- 命令上下文组装。
- 流程上下文组装。
- 故障上下文组装。
- 证据打包。

现有 pipeline 继续作为后台加工能力存在。

推荐把系统理解为：

```text
Pipeline 是知识生产线。
Graph-RAG Backend 是知识使用服务。
Skill 是 Agent 使用说明书和工具封装。
Agent 是最终交互入口。
```

## 16. 总结

这个项目后续的重点不应只是“继续抽取更多本体”，而应转向“让 Agent 真正能用知识”。

对于云核心网业务，最优先的使用场景不是抽象推理，而是：

- 查命令。
- 查参数。
- 查配置流程。
- 查前置条件。
- 查故障处理。
- 查影响范围。
- 查证据来源。

Graph-RAG 的价值在于把这些知识从零散文档片段组织成 Agent 可使用的上下文包。

最终推荐架构是：

```text
Agent 层：使用已有 Agent 框架，负责自然语言交互和最终回答。

Skill 层：封装云核心网知识后端，定义工具、调用策略、答案格式和追问规则。

后端服务层：重点建设，负责文档加工、索引、Graph-RAG 检索、图扩展、重排序、上下文组装和证据返回。
```

最小可用版本应先聚焦：

```text
Stage 1/2
+ segment 向量索引
+ RST 上下文扩展
+ 命令检索
+ Agent 上下文包
+ Skill 工具封装
```

当这个闭环跑通后，再逐步引入完整本体、事实抽取、流程图、故障图和影响分析能力。
